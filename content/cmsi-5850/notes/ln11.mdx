## The Operational Intuition

What does `3 + 5 * 2` mean? One answer: watch what happens when you compute it.

1. First, compute `5 * 2` — that gives `10`.
2. Then, compute `3 + 10` — that gives `13`.
3. Done. The meaning is `13`.

This is the operational approach: meaning as **simulation**. We define a machine (even if it's abstract), load the program into it, and show what happens when it runs. The machine's behavior *is* the meaning.

> The word "operational" literally means "pertaining to operations." An operational semantics defines meaning by describing the operations that a machine performs. You've been doing this since you first traced through code with pencil and paper — operational semantics just makes that process mathematically precise.

There are multiple ways to formalize this intuition, ranging from very concrete (build an actual virtual machine) to very abstract (write inference rules). We'll briefly mention the concrete approach, then spend the lecture on the two formal approaches that matter most.

## Concrete Operational Semantics

One way to define meaning operationally is to **translate** the AST into instructions for a concrete machine. The most famous approach flattens the tree into a linear sequence of instructions for a **stack machine**.

<DefinitionBox term="Stack Machine">

A **stack machine** is an abstract computational device that uses a last-in-first-out (LIFO) stack as its primary data structure. Instructions operate by pushing values onto the stack or popping values to compute results. Many real virtual machines use this model, including the Java Virtual Machine (JVM) and Python's bytecode interpreter (CPython).

</DefinitionBox>

### Engineering COS from the Tree Grammar

Recall from LN 10 that Petal's tree grammar is the **API contract** that any semantic approach must implement. For concrete operational semantics, each production in the grammar tells us what machine instructions to emit:

| Tree Grammar Production | Stack Machine Instruction(s) |
|---|---|
| <InlineMath formula="n" /> (number literal) | `PUSH n` |
| <InlineMath formula="e_1 \oplus e_2" /> (binary op) | <InlineMath formula="T\llbracket e_1 \rrbracket \cdot T\llbracket e_2 \rrbracket \cdot \text{OP}" /> |
| <InlineMath formula="i" /> (identifier) | `LOAD i` |
| <InlineMath formula="i\text{=}e" /> (assignment) | <InlineMath formula="T\llbracket e \rrbracket \cdot \text{STORE } i" /> |
| <InlineMath formula="\textbf{print}\; e" /> (print) | <InlineMath formula="T\llbracket e \rrbracket \cdot \text{PRINT}" /> |

The translation function <InlineMath formula="T" /> is compositional: to translate a compound node, translate its children first, then append the node's own instruction. This is a **post-order traversal** of the AST — the same structural recursion we keep seeing.

<DisplayMath formula="T\llbracket n \rrbracket = \text{PUSH } n \qquad T\llbracket e_1 + e_2 \rrbracket = T\llbracket e_1 \rrbracket \cdot T\llbracket e_2 \rrbracket \cdot \text{ADD}" />

The key insight: the **tree structure of the AST has been flattened** into a linear sequence. The translation tab makes this visible — the post-order traversal visits leaves before parents, ensuring operands are pushed before the operator that consumes them. The stack reconstructs the nesting at runtime.

> The JVM is one of the most successful stack machines ever built. Every Java, Kotlin, Scala, and Clojure program compiles down to JVM bytecode — a sequence of stack instructions not unlike the `PUSH`/`ADD` above, just with hundreds more operations. The operational semantics of the JVM is defined by the [JVM Specification](https://docs.oracle.com/javase/specs/jvms/se21/html/), a 600+ page document that describes every instruction's effect on the stack, heap, and execution state.

### The Register Machine Alternative

Stack machines aren't the only way to build a concrete machine. **Register machines** use a fixed set of named storage locations (registers) instead of an anonymous stack. The translation function changes fundamentally: instead of a simple post-order traversal with implicit operands, the translator must now **allocate a fresh register** for each intermediate result and **reference child registers by name** in parent instructions.

<DefinitionBox term="Register Machine">

A **register machine** is an abstract computational device that uses a fixed set of named registers (fast temporary storage locations) as its primary workspace. Instructions specify operands explicitly by register name, rather than relying on stack position. Most physical CPUs (x86, ARM, RISC-V) and modern compiler intermediate representations (LLVM IR) use this model.

</DefinitionBox>

Compare the two translation approaches: the stack machine used a simple post-order traversal with no bookkeeping — operands land on the stack implicitly. The register machine required the translator to **track which register holds each sub-expression's result** and **pass register names upward** to parent nodes. This is fundamentally more complex, but it has an advantage: register values **persist** after use. In the stack machine, values disappear the moment they're popped; in the register machine, `r1` still holds `3` even after the `ADD` consumed it. This makes register machines easier to debug and optimize — but harder to generate code for (register allocation is NP-hard in the general case).

| Feature | Stack Machine | Register Machine |
|---|---|---|
| **Operands** | Implicit (top of stack) | Explicit (named registers) |
| **Instruction size** | Smaller (no operand names) | Larger (register names in every instruction) |
| **Temporary values** | Pushed/popped automatically | Allocated to named registers |
| **Value lifetime** | Consumed on use (pop) | Persist until overwritten |
| **Real-world examples** | JVM, CPython, WebAssembly (MVP) | x86, ARM, LLVM IR, Dalvik (Android) |
| **Code generation** | Simpler (post-order traversal) | More complex (register allocation needed) |

> The choice between stack and register architectures is one of the deepest pragmatic decisions in language implementation. The JVM chose a stack architecture for portability (no assumptions about how many registers the host CPU has). LLVM chose a register architecture for performance (it maps more naturally to real hardware). Both are "correct" — they define the same meaning for the same programs. The difference is engineering, not semantics.

This approach is valuable (it's essentially how real VMs and CPUs work), but for our purposes, two more abstract formulations are more powerful. They let us reason about programs without committing to a particular machine architecture.

## Structural Operational Semantics (SOS) — Small-Step

The first formal approach was developed by Gordon Plotkin in 1981. The idea: define a **transition relation** that describes how a program configuration changes in a single step of computation.

<DefinitionBox term="Configuration">

A **configuration** is a snapshot of the machine's state at a single point in time. For Petal, a configuration is a pair $\langle e, \sigma \rangle$ where $e$ is an expression (or statement) and $\sigma$ is the current state (environment).

</DefinitionBox>

<DefinitionBox term="Small-Step (Structural) Operational Semantics">

**Small-step semantics** (also called **Structural Operational Semantics** or **SOS**) defines a **transition relation** $\to$ that describes one step of computation:

<DisplayMath formula="\langle e, \sigma \rangle \to \langle e', \sigma' \rangle" />

This reads: "In state $\sigma$, expression $e$ reduces in one step to $e'$, producing new state $\sigma'$."

A program executes by taking steps until no more transitions apply — the configuration is **terminal** (a final value).

</DefinitionBox>

> Gordon Plotkin's 1981 paper "A Structural Approach to Operational Semantics" is one of the most cited papers in programming language theory. His key innovation was making the rules *structural* — defined by induction on the syntax. This means you can determine which rule applies by looking at the top-level constructor of the expression. It's a direct consequence of compositionality: the step for a compound expression is determined by the steps of its parts.

The transition relation is defined by **inference rules** — exactly the kind of logical rules we studied in LN 2. Each rule has premises above the line and a conclusion below:

<DisplayMath formula="\frac{\text{premise}_1 \quad \text{premise}_2 \quad \cdots}{\text{conclusion}}" />

<DefinitionBox term="Inference Rule (Semantic)">

A **semantic inference rule** specifies one step of evaluation. The premises (above the line) describe conditions that must hold; the conclusion (below the line) describes the step that follows. Rules with no premises are called **axioms** — they apply unconditionally.

</DefinitionBox>

### Reading Inference Rules: The Omega Technique

Inference rules look intimidating at first — a fraction with Greek letters, angle brackets, and arrows. But there's a simple, repeatable technique for reading any rule. Trace an **Omega (Ω)** path through it:

1. **Start at the bottom-left** of the conclusion — this is what you *have* (the current configuration).
2. **Loop up** through each premise above the line — these are the conditions you must check or sub-evaluations you must perform.
3. **Come back down to the bottom-right** of the conclusion — this is what you *get* (the resulting value or new configuration).

The shape of this path — bottom-left, up, back down to bottom-right — traces an **Ω**. It works for every inference rule in this course. Let's practice it on the three small-step rules for binary operations — each covers a different case:

#### Why ⊕? The Metavariable Trick

You may have noticed that the rules above use <InlineMath formula="\oplus" /> rather than a specific operator like `+` or `*`. That's because <InlineMath formula="\oplus" /> is a **metavariable** — it stands for *any* of Petal's six binary operators (`+`, `-`, `*`, `/`, `%`, `**`). The side condition <InlineMath formula="v_1 \oplus v_2 = v" /> is the bridge: it says "interpret this syntactic operator as its corresponding mathematical operation and compute the result."

Without the metavariable, we would need to write three rules for *each* operator — left-operand steps, right-operand steps, and compute. Just the compute rules alone would look like:

<DisplayMath formula="\frac{v_1 + v_2 = v}{\langle v_1 + v_2, \sigma \rangle \to \langle v, \sigma \rangle} \quad \frac{v_1 - v_2 = v}{\langle v_1 - v_2, \sigma \rangle \to \langle v, \sigma \rangle} \quad \frac{v_1 \times v_2 = v}{\langle v_1 \times v_2, \sigma \rangle \to \langle v, \sigma \rangle} \quad \cdots" />

That's 6 operators × 3 rules each = **18 rules** for binary operations alone. The metavariable <InlineMath formula="\oplus" /> collapses all 18 into just 3 parametric rules — one rule *family* that covers every operator. This is standard practice in formal semantics: whenever a set of rules differs only in the operator symbol, factor it out with a metavariable and let a side condition do the dispatch.

### Engineering SOS Rules from the Tree Grammar

Before we present the full rule set, let's derive one rule from scratch. This exercise shows that the rules are not arbitrary — they are **mechanically determined** by the tree grammar. Consider the binary operation production <InlineMath formula="e_1 \oplus e_2" />:

**Step 1: Identify the node.** The tree grammar tells us the AST has a binary operator node with two children.

**Step 2: Identify the children.** The grammar says it has two sub-expressions, <InlineMath formula="e_1" /> and <InlineMath formula="e_2" />.

**Step 3: Formulate the small-step strategy.** SOS reduces *one* sub-expression per step. We must choose an evaluation order. We choose **left-to-right**. This gives us three cases:

1. If <InlineMath formula="e_1" /> is not yet a value, reduce it — the whole expression takes a step by reducing its left child.
2. If <InlineMath formula="e_1" /> is a value but <InlineMath formula="e_2" /> is not, reduce <InlineMath formula="e_2" /> — the whole expression takes a step by reducing its right child.
3. If both are values, apply the operator to get the final value.

**Step 4: Write the rules.** Each case becomes one inference rule:

<DisplayMath formula="\frac{\langle e_1, \sigma \rangle \to \langle e_1', \sigma \rangle}{\langle e_1 \oplus e_2, \sigma \rangle \to \langle e_1' \oplus e_2, \sigma \rangle} \quad \frac{\langle e_2, \sigma \rangle \to \langle e_2', \sigma \rangle}{\langle v_1 \oplus e_2, \sigma \rangle \to \langle v_1 \oplus e_2', \sigma \rangle} \quad \frac{v_1 \oplus v_2 = v}{\langle v_1 \oplus v_2, \sigma \rangle \to \langle v, \sigma \rangle}" />

Three rules from one grammar production. The tree grammar told us *what* to handle; the evaluation-order decision told us *how many* rules we need. A right-to-left order would produce three different rules. A parallel evaluation strategy would produce yet another set. The grammar constrains; the semanticist chooses.

> This is the key insight: you don't *memorize* inference rules. You **derive** them from the syntax. The tree grammar is the API; the inference rules are the implementation. Every production becomes one or more rules, and the structure of the production tells you what the premises must be.

### Rules for Petal Expressions

Here are the small-step rules for Petal's expression language:

**Numerals** — map the syntactic numeral to a mathematical number:

<DisplayMath formula="\frac{\mathcal{N}\llbracket \overline{n} \rrbracket = n}{\langle \overline{n}, \sigma \rangle \to \langle n, \sigma \rangle}" />

This rule deserves attention because it reveals something fundamental. The <InlineMath formula="\overline{n}" /> on the left is a **numeral** — a syntactic object, a sequence of digit characters sitting in the AST. The <InlineMath formula="n" /> on the right is a **number** — a mathematical object living in the semantic domain of values. They are *not* the same thing. The string `"42"` and the integer 42 belong to entirely different worlds.

The valuation function <InlineMath formula="\mathcal{N}\llbracket \cdot \rrbracket" /> is the bridge between them. It interprets a syntactic numeral as its mathematical value — the same kind of syntax-to-semantics mapping that <InlineMath formula="\sigma(x)" /> performs for identifiers and <InlineMath formula="v_1 \oplus v_2 = v" /> performs for operators. In fact, every leaf node in the AST has such a bridge:

| AST Leaf | Bridge | From (Syntax) | To (Semantics) |
|----------|--------|----------------|-----------------|
| Numeral <InlineMath formula="\overline{n}" /> | <InlineMath formula="\mathcal{N}\llbracket \overline{n} \rrbracket = n" /> | Digit string in the AST | Mathematical number |
| Identifier <InlineMath formula="x" /> | <InlineMath formula="\sigma(x) = v" /> | Name token in the AST | Value bound in the environment |
| Operator <InlineMath formula="\oplus" /> | <InlineMath formula="v_1 \oplus v_2 = v" /> | Operator terminal in the AST | Mathematical operation on values |

This is the job of semantics: translate *every* syntactic entity into its corresponding semantic domain. The inference rules make each translation explicit through side conditions and premises. Nothing crosses from syntax to meaning silently.

> Many textbooks elide this distinction and write <InlineMath formula="\langle n, \sigma \rangle \to \langle n, \sigma \rangle" /> — treating numeral and number as identical. That works when the mapping is "obvious" (the numeral `42` maps to 42), but it hides the conceptual step. We keep the valuation function explicit because it reinforces the pattern: *every* rule is a syntax-to-semantics translation, and every leaf in the tree grammar must have its own bridge.

**Identifiers** look up the environment:

<DisplayMath formula="\frac{\sigma(x) = v}{\langle x, \sigma \rangle \to \langle v, \sigma \rangle}" />

**Binary operations** — evaluate left operand first, then right, then apply:

<DisplayMath formula="\frac{\langle e_1, \sigma \rangle \to \langle e_1', \sigma \rangle}{\langle e_1 \oplus e_2, \sigma \rangle \to \langle e_1' \oplus e_2, \sigma \rangle}" />

<DisplayMath formula="\frac{\langle e_2, \sigma \rangle \to \langle e_2', \sigma \rangle}{\langle v_1 \oplus e_2, \sigma \rangle \to \langle v_1 \oplus e_2', \sigma \rangle}" />

<DisplayMath formula="\frac{v_1 \oplus v_2 = v}{\langle v_1 \oplus v_2, \sigma \rangle \to \langle v, \sigma \rangle}" />

Notice the left-to-right evaluation order: we reduce $e_1$ until it's a value, *then* reduce $e_2$. This is a design choice — we could have defined right-to-left or even parallel evaluation. The rules make the choice explicit.

> This is where formal semantics earns its keep. Remember from LN 10, the expression `(x = 2) + x` had different meanings depending on evaluation order. Small-step rules eliminate this ambiguity: the left operand is fully reduced before the right operand is touched. A different set of rules would give a different semantics — and both would be mathematically precise.

### Connection to LN 4

Here's the key insight: **you already know small-step operational semantics**. Beta reduction in the lambda calculus:

<DisplayMath formula="(\lambda x.\, e) \; v \to e[v/x]" />

This is a small-step transition rule. The lambda calculus reduction rules from LN 4 *are* structural operational semantics. You've been doing SOS since week four — you just didn't have the name for it yet.

The evaluation strategies from LN 4 also map directly:

| LN 4 Term | SOS Term |
|-----------|----------|
| Beta reduction | Transition step |
| Normal form | Terminal configuration |
| Call-by-value | Reduce arguments before applying |
| Call-by-name | Substitute arguments unreduced |
| Church-Rosser | Confluence of the transition relation |

> This connection is deep. The lambda calculus, developed by Church in the 1930s, was the first operational semantics — decades before the term existed. When Plotkin formalized SOS in 1981, he was generalizing what Church had already been doing for pure functions. Operational semantics extends the lambda calculus's reduction approach to handle assignment, state, control flow, and all the other features that separate programming languages from pure mathematics.

### Determinism and Non-Determinism

Our rules for Petal are **deterministic** — for any configuration, at most one rule applies. The left-to-right evaluation order ensures this: we always know which subexpression to reduce next. But SOS can also describe **non-deterministic** systems by providing multiple rules for the same configuration. This is essential for modeling concurrent programs, where multiple threads can take steps in any order.

## Natural Semantics — Big-Step

The second formal approach was developed by Gilles Kahn in 1987. Instead of showing each tiny step, big-step semantics defines how an expression evaluates to its **final value** in one judgment.

<DefinitionBox term="Big-Step (Natural) Semantics">

**Big-step semantics** (also called **Natural Semantics**) defines an **evaluation relation** $\Downarrow$ that relates a configuration to its final result:

<DisplayMath formula="\langle e, \sigma \rangle \Downarrow v" />

This reads: "In state $\sigma$, expression $e$ evaluates to value $v$."

The relation is defined by inference rules where the premises are themselves big-step evaluations.

</DefinitionBox>

> Gilles Kahn, who worked at INRIA in France, chose the name "natural semantics" because the inference rules feel natural — they describe what the programmer would say if asked "what does this expression evaluate to?" They also mirror the structure of natural deduction from LN 2.

### Engineering Big-Step Rules from the Tree Grammar

Let's repeat the engineering exercise from the SOS section, but now for big-step. Take the same production, <InlineMath formula="e_1 \oplus e_2" />:

**Step 1: Identify the node and children.** Same as before — a binary operator with two sub-expressions.

**Step 2: Formulate premises.** Big-step evaluates each child to its *final value* in one judgment. We don't need to specify evaluation order within a single rule — both children are evaluated to completion:

<DisplayMath formula="\langle e_1, \sigma \rangle \Downarrow v_1 \qquad \langle e_2, \sigma \rangle \Downarrow v_2 \qquad v_1 \oplus v_2 = v" />

**Step 3: Formulate conclusion.** Combine the results into a single judgment for the whole expression:

<DisplayMath formula="\langle e_1 \oplus e_2, \sigma \rangle \Downarrow v" />

**Step 4: Assemble the rule.**

<DisplayMath formula="\frac{\langle e_1, \sigma \rangle \Downarrow v_1 \quad \langle e_2, \sigma \rangle \Downarrow v_2 \quad v_1 \oplus v_2 = v}{\langle e_1 \oplus e_2, \sigma \rangle \Downarrow v}" />

**One rule** instead of three. Big-step is more compact because it doesn't specify evaluation order *within* the rule — each premise is an independent judgment that evaluates a sub-expression all the way to a value. The trade-off: big-step hides the intermediate steps, which makes it simpler to write but harder to use for debugging or reasoning about non-termination.

Now let's apply the Omega technique to a **statement** rule — the assignment <InlineMath formula="i\text{=}e" />. Notice how the configuration shape changes: statements transform the entire machine state (environment + output), not just expressions.

<InferenceRuleReader
  title="Reading the Assignment Rule"
  ruleName="Assignment (Big-Step)"
  conclusion={{
    left: {
      label: "Current configuration",
      latex: "\\langle x = e,\\; \\sigma,\\; o \\rangle",
      description: "Start here: you have an assignment statement 'x = e' in environment σ with current output o."
    },
    right: {
      label: "New configuration",
      latex: "\\Downarrow \\langle \\sigma[x \\mapsto v],\\; o \\rangle",
      description: "End here: the environment is updated — x now maps to v. The output o is unchanged (assignments don't print)."
    }
  }}
  premises={[
    {
      label: "Evaluate the right-hand side",
      latex: "\\langle e,\\; \\sigma \\rangle \\Downarrow v",
      description: "The only premise: evaluate the expression e in the current environment σ. It produces a value v. This is a recursive big-step judgment — it may itself require further evaluation."
    }
  ]}
/>

Compare this to the binary operation rule: the assignment has only **one premise** (evaluate the RHS) because it has only one sub-expression child. The tree grammar's structure directly determines the number of premises. One child, one premise. Two children, two premises (or more, if there are side conditions).

### Rules for Petal Expressions

**Numerals** — the valuation function maps syntax to value (same bridge as in SOS):

<DisplayMath formula="\frac{\mathcal{N}\llbracket \overline{n} \rrbracket = n}{\langle \overline{n}, \sigma \rangle \Downarrow n}" />

**Identifiers** look up the environment:

<DisplayMath formula="\frac{\sigma(x) = v}{\langle x, \sigma \rangle \Downarrow v}" />

**Binary operations** — evaluate both operands, then apply:

<DisplayMath formula="\frac{\langle e_1, \sigma \rangle \Downarrow v_1 \quad \langle e_2, \sigma \rangle \Downarrow v_2 \quad v_1 \oplus v_2 = v}{\langle e_1 \oplus e_2, \sigma \rangle \Downarrow v}" />

### Rules for Petal Statements

**Assignment** — evaluate the expression, update the environment:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow v}{\langle x = e, \sigma, o \rangle \Downarrow \langle \sigma[x \mapsto v], o \rangle}" />

**Print** — evaluate the expression, append to output:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow v}{\langle \text{print } e, \sigma, o \rangle \Downarrow \langle \sigma, o \cdot v \rangle}" />

**Sequential execution** — thread state through:

<DisplayMath formula="\frac{\langle s_1, \sigma, o \rangle \Downarrow \langle \sigma', o' \rangle \quad \langle s_2, \sigma', o' \rangle \Downarrow \langle \sigma'', o'' \rangle}{\langle s_1; s_2, \sigma, o \rangle \Downarrow \langle \sigma'', o'' \rangle}" />

> The sequencing rule is where **state threading** happens. Notice how $\sigma'$ appears as *output* of the first premise and *input* of the second. The state flows from one statement to the next like water through a pipeline. This is the formal counterpart of the imperative programmer's intuition that statements execute "in order" — the environment produced by one statement becomes the starting environment for the next.

### What Do Statements Look Like Small-Step by Step?

Having just seen the big-step rules for statements, you might wonder: what do these same constructs look like under the small-step lens? In the SOS section above, we only covered expressions. Let's complete the picture by deriving the small-step rules for statements.

The key difference: a statement configuration includes the output stream — $\langle s, \sigma, o \rangle$ — and a completed statement produces a terminal configuration $\langle \sigma', o' \rangle$ (no remaining statement to execute). Each rule follows the same pattern we saw with expressions: if a sub-expression can step, step it; if everything is a value, perform the side effect.

**Assignment** — reduce the RHS expression, then bind:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \to \langle e', \sigma \rangle}{\langle x\text{=}e, \sigma, o \rangle \to \langle x\text{=}e', \sigma, o \rangle}" />

<DisplayMath formula="\frac{}{\langle x\text{=}v, \sigma, o \rangle \to \langle \sigma[x \mapsto v], o \rangle}" />

The first rule steps the right-hand side expression. The second is an axiom: once the RHS is a value, the assignment completes by updating the environment. Compare this to the big-step version — big-step collapses both into a single judgment, while small-step separates "evaluate the expression" from "perform the binding."

**Print** — reduce the argument expression, then emit:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \to \langle e', \sigma \rangle}{\langle \text{print } e, \sigma, o \rangle \to \langle \text{print } e', \sigma, o \rangle}" />

<DisplayMath formula="\frac{}{\langle \text{print } v, \sigma, o \rangle \to \langle \sigma, o \cdot v \rangle}" />

Same pattern: step the argument until it's a value, then append to the output stream.

**Sequential execution** — step the first statement, then continue:

<DisplayMath formula="\frac{\langle s_1, \sigma, o \rangle \to \langle s_1', \sigma', o' \rangle}{\langle s_1;\, s_2, \sigma, o \rangle \to \langle s_1';\, s_2, \sigma', o' \rangle}" />

<DisplayMath formula="\frac{\langle s_1, \sigma, o \rangle \to \langle \sigma', o' \rangle}{\langle s_1;\, s_2, \sigma, o \rangle \to \langle s_2, \sigma', o' \rangle}" />

The first rule says: if the first statement can take a step (but isn't done yet), the whole sequence takes a step. The second rule says: when the first statement completes (reaches a terminal configuration), discard it and move to the second statement with the updated state. This is where you can see the state threading happen *one step at a time*, compared to big-step's all-at-once approach.

> Notice the structural parallel: each big-step statement rule has a corresponding *pair* of SOS rules — one for "the sub-expression is still reducing" and one for "the sub-expression is done, perform the effect." Big-step merges these into a single rule with a recursive premise. This is the fundamental trade-off: big-step is more compact, small-step shows more detail.

## Small-Step vs. Big-Step

| Small-Step (SOS) | Big-Step (Natural) |
|---|---|
| Shows each tiny reduction | Shows final result directly |
| One step at a time: $\to$ | One big evaluation: $\Downarrow$ |
| Can express non-termination | Cannot directly express non-termination |
| Can express interleaving/concurrency | Cannot express interleaving |
| More rules, more detail | Fewer rules, more compact |
| Gordon Plotkin, 1981 | Gilles Kahn, 1987 |

Small-step is more powerful: it can describe programs that don't terminate (they just keep stepping forever) and concurrent programs (interleave steps from different threads). Big-step is more convenient: it's closer to how we think about evaluation, and it maps directly to a recursive evaluator.

> An important practical difference: when a program has a bug, small-step semantics lets you inspect the *exact step* where things went wrong. Big-step can only tell you that the entire evaluation failed. This is why debuggers (like `gdb` or Chrome DevTools) work in a small-step style — they let you "step through" one instruction at a time.

## Extending Petal

Petal so far has only expressions, assignments, and print. To make semantics interesting, we need **control flow**. But before we dive in, notice what adding a new feature to a language actually requires — it activates the **entire pipeline** we've been building across this course:

1. **Define the concrete syntax** (LN 7–8) — what does the new construct look like as source text?
2. **Update the grammar** (LN 7) — add productions to the CFG so the parser can recognize it.
3. **Update the tree grammar** (LN 8) — add new AST node shapes so the semantic phase knows what to handle.
4. **Define the semantic rules** (LN 11–13) — derive inference rules, equations, or axioms from the new tree grammar nodes.

And if the new feature relies on mathematics not yet in our toolkit — imagine a hypothetical `integrate` keyword that computes definite integrals, or a `sample` expression drawing from probability distributions — we would need to go even further back to the foundations (LN 1–4) and extend our mathematical tools before any of the above steps are possible.

For if-else and while, we don't need new math — the existing tools (booleans encoded as integers, recursion from LN 4) suffice. So we start at step 3: updating the tree grammar. The original `Stmt` production from LN 8 had only two forms:

<DisplayMath formula="s : \text{Stmt} = i\text{=}e \mid \textbf{print}\; e" />

We extend it with two new productions:

<DisplayMath formula="s : \text{Stmt} = i\text{=}e \mid \textbf{print}\; e \mid \textbf{if}\; e\; \textbf{then}\; s^+\; \textbf{else}\; s^+ \mid \textbf{while}\; e\; \textbf{do}\; s^+" />

Both new constructs are statements, not expressions. Each has an expression child (the condition) and one or two statement-list children (the branches or body). The `Exp` and `Program` productions remain unchanged — only `Stmt` grows. Now the tree grammar tells us exactly what rules we must write.

### If-Else

From the tree grammar: <InlineMath formula="\textbf{if}\; e\; \textbf{then}\; s^+\; \textbf{else}\; s^+" /> — one expression child (the condition) and two statement-list children (the branches). This structure tells us we need two rules: one for when the condition is truthy, one for when it's falsy.

```
if x > 0 {
  y = x;
} else {
  y = 0;
}
```

The AST node:
```json
{ "type": "IfElse", "condition": Exp, "then": Statement[], "else": Statement[] }
```

<DefinitionBox term="Conditional Semantics">

The semantics of a conditional is defined by **two rules** — one for each branch. The condition expression is evaluated, and its value determines which branch executes:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow v \quad v \neq 0 \quad \langle s_1, \sigma, o \rangle \Downarrow \langle \sigma', o' \rangle}{\langle \textbf{if}\; e\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle \Downarrow \langle \sigma', o' \rangle}" />

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow 0 \quad \langle s_2, \sigma, o \rangle \Downarrow \langle \sigma', o' \rangle}{\langle \textbf{if}\; e\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle \Downarrow \langle \sigma', o' \rangle}" />

</DefinitionBox>

We use the C convention: any non-zero value is "truthy," zero is "falsy." This is a pragmatic choice (LN 6).

#### Small-Step (SOS) Rules

The big-step rules above collapse the entire conditional into one judgment. The small-step version makes the condition reduction explicit — each sub-expression in `e` steps individually before the branch is chosen. We need three rules:

<DefinitionBox term="Conditional — Small-Step Rules">

**Reduce the condition** — if `e` can still step, the whole if-else steps by stepping `e`:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \to \langle e', \sigma \rangle}{\langle \textbf{if}\; e\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle \to \langle \textbf{if}\; e'\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle}" />

**Condition is truthy** — once `e` has reduced to a value $v \neq 0$, take the then-branch:

<DisplayMath formula="\frac{v \neq 0}{\langle \textbf{if}\; v\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle \to \langle s_1,\; \sigma,\; o \rangle}" />

**Condition is falsy** — if the value is $0$, take the else-branch:

<DisplayMath formula="\frac{}{\langle \textbf{if}\; 0\; \textbf{then}\; s_1\; \textbf{else}\; s_2,\; \sigma,\; o \rangle \to \langle s_2,\; \sigma,\; o \rangle}" />

</DefinitionBox>

Compare the two styles: big-step says "evaluate the condition to a value, then run one branch." Small-step says "keep stepping the condition until it's a value, *then* pick a branch." The big-step version hides the intermediate computation inside the premise <InlineMath formula="\langle e, \sigma \rangle \Downarrow v" />; the small-step version exposes every individual reduction step on the condition expression.

> The conditional is where semantics first diverges from pure expression evaluation. Expressions always produce values; conditionals produce *different effects* depending on runtime data. This is the fundamental mechanism that makes programs interesting — without conditionals, every program is a straight-line calculation.

Notice something important: the unused branch ($s_2$ in the truthy case, $s_1$ in the falsy case) is **never evaluated**. This is not just an optimization — it's a semantic decision. The meaning of the conditional depends on only one branch. If the other branch contains an error (like division by zero), it doesn't matter — it's never reached. This connects to LN 10's discussion of short-circuit evaluation: the language deliberately avoids evaluating expressions that aren't needed.

### Comparison: Conditionals Across Languages

Different languages express the same semantic pattern in syntactically distinct ways:

```python
# Python: conditional expression (ternary)
y = x if x > 0 else 0
```

```javascript
// JavaScript: ternary operator
const y = x > 0 ? x : 0;
```

```haskell
-- Haskell: if-then-else is an expression (always returns a value)
let y = if x > 0 then x else 0
```

```rust
// Rust: if is also an expression
let y = if x > 0 { x } else { 0 };
```

In Haskell and Rust, `if` is an *expression* — it returns a value. In Python and JavaScript, the ternary form is an expression, but the statement form (`if ... else ...`) is a statement. Petal's if-else is a statement (it doesn't return a value). This is a **semantic design decision** that shapes how programmers think about control flow.

### While Loops

From the tree grammar: <InlineMath formula="\textbf{while}\; e\; \textbf{do}\; s^+" /> — one expression child (the condition) and one statement-list child (the body). The self-referential nature of loops means the continuation rule will reference the `while` construct itself in its own premise.

```
while x > 0 {
  x = x - 1;
}
```

The AST node:
```json
{ "type": "While", "condition": Exp, "body": Statement[] }
```

<DefinitionBox term="While Loop Semantics">

The semantics of a `while` loop requires **two rules** — one for termination and one for continuation:

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow 0}{\langle \textbf{while}\; e\; \textbf{do}\; s,\; \sigma,\; o \rangle \Downarrow \langle \sigma, o \rangle} \quad \text{(loop ends)}" />

<DisplayMath formula="\frac{\langle e, \sigma \rangle \Downarrow v \quad v \neq 0 \quad \langle s, \sigma, o \rangle \Downarrow \langle \sigma', o' \rangle \quad \langle \textbf{while}\; e\; \textbf{do}\; s,\; \sigma',\; o' \rangle \Downarrow \langle \sigma'', o'' \rangle}{\langle \textbf{while}\; e\; \textbf{do}\; s,\; \sigma,\; o \rangle \Downarrow \langle \sigma'', o'' \rangle} \quad \text{(loop continues)}" />

</DefinitionBox>

Notice something remarkable about the "loop continues" rule: the conclusion mentions `while e do s`, and so does one of the premises. **The rule refers to itself.** This is allowed in operational semantics because we're defining a relation, not a function — the rule says "if the body produces a new state, and *then* the whole loop produces a final state, then the loop produces that final state." It unfolds naturally through recursion.

> This self-reference will cause problems when we try denotational semantics in LN 12, because there we need compositionality — the meaning of the whole must be *defined* from the meanings of the parts, with no circularity. That's where fixed points and the Y combinator (LN 4) will come back, solving the same problem at a different level of abstraction.

#### Small-Step (SOS) Rules

The standard SOS approach for loops is the **unfolding rule**: on every step, the while construct rewrites itself into a conditional that either executes one iteration and re-enters the loop, or terminates. This requires one new primitive — <InlineMath formula="\textbf{skip}" />, a no-op statement that immediately completes without changing the state.

<DefinitionBox term="While Loop — Small-Step Rules">

**Unfolding** — the loop rewrites into a conditional:

<DisplayMath formula="\frac{}{\langle \textbf{while}\; e\; \textbf{do}\; s,\; \sigma,\; o \rangle \to \langle \textbf{if}\; e\; \textbf{then}\; (s;\; \textbf{while}\; e\; \textbf{do}\; s)\; \textbf{else}\; \textbf{skip},\; \sigma,\; o \rangle}" />

**Skip** — the no-op statement completes immediately:

<DisplayMath formula="\frac{}{\langle \textbf{skip},\; \sigma,\; o \rangle \to \langle \sigma,\; o \rangle}" />

</DefinitionBox>

The unfolding rule is an axiom — no premises needed. It converts the `while` into constructs we already have rules for: if-else and sequencing. After unfolding, the existing if-else SOS rules take over to evaluate the condition and choose a branch. If the condition is truthy, the then-branch executes the body `s` followed by the whole loop again (via sequencing). If the condition is falsy, `skip` fires and the loop terminates.

This reveals something deep: **loops are syntactic sugar for conditional recursion.** The while construct doesn't introduce any new reduction machinery — it simply rewrites into patterns we already know how to handle. This connects directly back to LN 4's fixed points: a loop is a fixed point of the transformation "check the condition, maybe do the body, then repeat." The SOS unfolding makes this structure explicit, while the big-step version hides it inside the self-referential premise.

The `skip` statement is trivial but necessary — it provides the "loop ends" case with a concrete statement to step to when the condition is falsy. Without it, we'd have no way to express "do nothing" as a statement in the SOS framework.

### What About the Concrete Machines?

We've seen how if-else and while work in big-step and small-step abstract semantics. But what do they look like when compiled to our stack machine from earlier in this lecture? Control flow at the machine level is fundamentally **program counter manipulation** — jumps, not rule selection. Two new instructions make this possible: `JUMPZ label` (jump to `label` if the top of the stack is zero, popping the value) and `JUMP label` (unconditional jump).

**If-else compilation:**

```
[compile e]          // push condition value
JUMPZ else_label     // if 0, jump to else branch
[compile s₁]         // then-branch instructions
JUMP end_label       // skip over else branch
else_label:
[compile s₂]         // else-branch instructions
end_label:
```

The condition expression compiles normally, leaving a value on the stack. `JUMPZ` pops it and branches: if zero, execution jumps to the else-branch; otherwise it falls through to the then-branch. The unconditional `JUMP` at the end of the then-branch skips over the else-branch.

**While loop compilation:**

```
loop_label:
[compile e]          // push condition value
JUMPZ end_label      // if 0, exit loop
[compile s]          // body instructions
JUMP loop_label      // go back to condition check
end_label:
```

Loops become **backward jumps**. The `JUMP loop_label` at the bottom sends the program counter back to the condition check, creating the repetition. When the condition evaluates to zero, `JUMPZ` breaks out of the loop by jumping forward past the body.

Notice the contrast with SOS: the small-step approach *unfolds* a while into a conditional (conditional recursion), while the COS approach compiles it into a backward branch (an explicit loop in the instruction stream). These are two very different implementation strategies for the same semantic behavior. The SOS approach stays close to the mathematical definition; the COS approach stays close to how real CPUs work — every `if` in source code becomes a conditional branch instruction, and every `while` becomes a backward branch. The register machine version would look similar, with register operands for the condition check instead of implicit stack values.

## From Rules to Code: The Interpreter

Here is the payoff. Every big-step inference rule maps directly to a clause in a recursive JavaScript function. The rules *are* the specification; the code *is* the implementation.

<DefinitionBox term="Tree-Walking Interpreter">

A **tree-walking interpreter** evaluates a program by recursively traversing the AST, visiting each node and computing its meaning directly. Each node type corresponds to a function clause that implements the matching inference rule.

</DefinitionBox>

```javascript
function evaluate(node, env) {
  switch (node.type) {
    case "Number":
      return node.value;

    case "Identifier":
      if (!(node.name in env)) throw new Error(`Undefined: ${node.name}`);
      return env[node.name];

    case "BinaryOp": {
      const left = evaluate(node.left, env);
      const right = evaluate(node.right, env);
      switch (node.op) {
        case "+": return left + right;
        case "-": return left - right;
        case "*": return left * right;
        case "/": return left / right;
        case "**": return left ** right;
        default: throw new Error(`Unknown op: ${node.op}`);
      }
    }

    case "Assignment":
      env[node.name] = evaluate(node.value, env);
      return;

    case "Print": {
      const value = evaluate(node.expression, env);
      console.log(value);
      return;
    }

    case "IfElse": {
      const cond = evaluate(node.condition, env);
      const branch = cond !== 0 ? node.then : node.else;
      for (const stmt of branch) evaluate(stmt, env);
      return;
    }

    case "While":
      while (evaluate(node.condition, env) !== 0) {
        for (const stmt of node.body) evaluate(stmt, env);
      }
      return;

    case "Program":
      for (const stmt of node.body) evaluate(stmt, env);
      return;
  }
}
```

Compare each `case` to the corresponding inference rule:

| Rule | Code |
|------|------|
| $\mathcal{N}\llbracket \overline{n} \rrbracket = n \implies \langle \overline{n}, \sigma \rangle \Downarrow n$ | `case "Number": return node.value` |
| $\sigma(x) = v \implies \langle x, \sigma \rangle \Downarrow v$ | `case "Identifier": return env[node.name]` |
| $\langle e_1, \sigma \rangle \Downarrow v_1, \; \langle e_2, \sigma \rangle \Downarrow v_2$ | `const left = evaluate(node.left, env); const right = ...` |
| $\langle x = e, \sigma \rangle \Downarrow \sigma[x \mapsto v]$ | `env[node.name] = evaluate(node.value, env)` |

The correspondence is one-to-one. Every premise in a rule becomes a recursive call in the code. Every conclusion becomes a return value or state update. The inference rules are not just mathematics — they are **pseudocode for the interpreter**.

> This is not a coincidence. Plotkin designed SOS so that its rules would be directly implementable. The "structural" part of Structural Operational Semantics means the rules are defined by induction on the structure of the syntax — and induction on structure is exactly what a recursive function does.

This is the first implementation of Petal's meaning. Given an AST (the output of the parser from LN 9), this function computes the program's result. We've gone from raw text to meaning:

```
Source → Tokens → CST → AST → evaluate(ast, {}) → Output
                                    ^^^^^^^^^^^^^^^
                                    Today's contribution
```

### The Same Pattern in Other Languages

The tree-walking interpreter pattern is universal. Here's the same `evaluate` function in Python and Haskell:

```python
def evaluate(node, env):
    match node["type"]:
        case "Number":
            return node["value"]
        case "Identifier":
            return env[node["name"]]
        case "BinaryOp":
            left = evaluate(node["left"], env)
            right = evaluate(node["right"], env)
            match node["op"]:
                case "+": return left + right
                case "*": return left * right
        case "Assignment":
            env[node["name"]] = evaluate(node["value"], env)
        case "Print":
            print(evaluate(node["expression"], env))
        case "Program":
            for stmt in node["body"]:
                evaluate(stmt, env)
```

```haskell
evaluate :: AST -> Env -> (Value, Env)
evaluate (Num n) env       = (n, env)
evaluate (Var x) env       = (env ! x, env)
evaluate (BinOp op l r) env =
  let (vl, _) = evaluate l env
      (vr, _) = evaluate r env
  in (applyOp op vl vr, env)
evaluate (Assign x e) env =
  let (v, _) = evaluate e env
  in (v, insert x v env)
```

> Notice how Haskell's pattern matching mirrors the inference rules even more closely than JavaScript's `switch`. Each clause in the Haskell version IS a rule: the pattern on the left is the configuration, the expression on the right is the conclusion. Haskell was designed by programming language theorists — its syntax was crafted to look like mathematical notation.

## Interpreters in the Wild

The tree-walking interpreter pattern is not a toy — it's how many real languages work:

import ast

tree = ast.parse("x = 3 + 5")

```

// Babel visitors implement semantic actions on each node type
// — the same pattern as our evaluate() switch statement
```

```rust

// and borrow checking — both are semantic analyses
// that follow inference rules, just like our evaluator
```

### Historical Examples

| Language | Interpreter Style | Notes |
|----------|------------------|-------|
| **Early Ruby (MRI)** | Tree-walking | Ruby 1.8 directly walked the AST; Ruby 1.9+ compiles to bytecode |
| **R** | Tree-walking | R's interpreter still walks the AST today |
| **Bash/Shell** | Tree-walking | Shell scripts are parsed and evaluated node by node |
| **Python (CPython)** | Bytecode + VM | Compiles AST to stack machine instructions (the concrete approach) |
| **Java (JVM)** | Bytecode + JIT | Compiles to stack bytecode, then JIT-compiles hot paths to native |

> Robert Nystrom, in *Crafting Interpreters*, calls the tree-walking interpreter "the simplest, most obvious implementation technique." He also notes its limitation: performance. Walking a tree involves pointer chasing (following references from node to node), which is slow on modern CPUs that prefer sequential memory access. That's why production languages typically compile to bytecode — but the semantics are the same.

## Proof Trees

Inference rules don't just define evaluation — they also let us **prove** that a specific program produces a specific result.

<DefinitionBox term="Proof Tree (Derivation Tree)">

A **proof tree** (or **derivation tree**) is a tree-shaped composition of inference rules that proves a specific evaluation judgment. Each node in the tree is a judgment, each edge is a rule application, and the leaves are axioms (rules with no premises).

</DefinitionBox>

For the program `x = 3 + 5; print x;`, the proof tree for the first statement looks like:

<DisplayMath formula="\frac{\displaystyle\frac{}{\langle 3, \sigma \rangle \Downarrow 3} \quad \frac{}{\langle 5, \sigma \rangle \Downarrow 5} \quad 3 + 5 = 8}{\displaystyle\frac{\langle 3 + 5, \sigma \rangle \Downarrow 8}{\langle x = 3 + 5, \sigma, o \rangle \Downarrow \langle \sigma[x \mapsto 8], o \rangle}}" />

The tree reads bottom-up: to prove that `x = 3 + 5` produces the binding $x \mapsto 8$, we need to prove that `3 + 5` evaluates to `8`, which requires proving that `3` evaluates to `3` and `5` evaluates to `5` (both are axioms — no premises needed).

This connects directly to LN 2: inference rules ARE logical deduction. A proof tree for a program evaluation is structurally identical to a proof tree in predicate logic. The rules are the axioms and inference rules of a logical system; the proof tree is a derivation in that system.

### Worked Example: A Larger Proof Tree

Let's build the proof tree for `y = (2 + 3) * 4`:

<DisplayMath formula="\frac{\displaystyle\frac{\displaystyle\frac{}{\langle 2, \sigma \rangle \Downarrow 2} \quad \frac{}{\langle 3, \sigma \rangle \Downarrow 3} \quad 2 + 3 = 5}{\langle 2 + 3, \sigma \rangle \Downarrow 5} \quad \frac{}{\langle 4, \sigma \rangle \Downarrow 4} \quad 5 \times 4 = 20}{\displaystyle\frac{\langle (2+3) \times 4, \sigma \rangle \Downarrow 20}{\langle y = (2+3) \times 4, \sigma, o \rangle \Downarrow \langle \sigma[y \mapsto 20], o \rangle}}" />

The tree structure mirrors the AST structure — each AST node corresponds to one judgment in the proof tree. This is compositionality made visible: the proof of the whole is assembled from proofs of the parts.

> Notice that proof trees and ASTs have a deep structural similarity. Both are trees. Both are defined by induction on their structure. Both decompose a complex entity into simpler parts. This is not a coincidence — it's a consequence of the fact that syntax (ASTs) and semantics (proof trees) are both defined by structural induction, which is itself grounded in the well-foundedness of the natural numbers (LN 3).

### Non-Termination and Proof Trees

What about `while 1 { print 0; }`? The proof tree for this program would require:

1. Prove $\langle 1, \sigma \rangle \Downarrow 1$ (axiom — fine)
2. $1 \neq 0$, so use the "loop continues" rule
3. Execute the body: $\langle \text{print } 0, \sigma, o \rangle \Downarrow \langle \sigma, o \cdot 0 \rangle$
4. Now prove the loop again from step 1, with $o' = o \cdot 0$

The proof tree is **infinitely deep** — it never reaches the base case because the condition never becomes 0. This is big-step semantics' limitation: infinite computation produces an infinitely large proof tree, which is not a valid finite proof. Big-step semantics simply cannot express non-termination.

Small-step semantics handles this naturally: the transition sequence <InlineMath formula="$\langle \text{while} \ldots \rangle \to \langle \ldots \rangle \to \langle \text{while} \ldots \rangle \to \cdots$" /> continues forever. It's an infinite *sequence* of finite steps, which is perfectly well-defined mathematically.

> This asymmetry between big-step and small-step semantics is why language specifications that need to describe non-terminating behavior (like the semantics of concurrent programming languages) almost always use small-step. The JavaScript specification (ECMAScript), for example, uses a small-step style that can describe both terminating and non-terminating programs.

### When Proof Trees Get Stuck

What happens when we try to build a proof tree for an erroneous program?

```
print x;
```

We need to prove $\langle x, \sigma_0 \rangle \Downarrow v$ where $\sigma_0$ is the empty environment. The identifier rule requires $\sigma_0(x) = v$, but $x$ is not in the empty environment. **No rule applies.** The proof tree cannot be completed.

<DefinitionBox term="Stuck Configuration">

A **stuck configuration** is one where the expression is not a value but no transition rule applies. In operational semantics, stuck configurations represent runtime errors — type mismatches, undeclared variables, division by zero. The absence of a rule IS the error handling.

</DefinitionBox>

This is how operational semantics handles errors: the computation "gets stuck." There is no transition that applies, so execution cannot proceed. The program has no meaning under the operational semantics — which is exactly right, because evaluating an undeclared variable *should* be meaningless.

> There is an elegant duality here: well-typed programs never get stuck (this is the **progress theorem** from type theory), and stuck programs are always ill-typed. Type systems, which we'll explore in LN 14, exist precisely to ensure at compile time that no stuck configurations can occur at runtime. This is the bridge from operational semantics to axiomatic semantics — static analysis prevents dynamic errors.

## Summary

Today we formalized the first approach to semantics: **operational semantics — meaning by running**. We developed two styles:

**Small-step (SOS)**: Plotkin's transition relation shows each individual reduction. This is the same mechanism as beta reduction in the lambda calculus (LN 4) — you've been doing operational semantics since week four.

**Big-step (Natural)**: Kahn's evaluation relation shows how an expression evaluates to its final value in one judgment. This is more compact and maps directly to a recursive evaluator.

We extended Petal with **if-else** and **while** loops, writing operational rules for each. We noted the self-referential while rule — it works in operational semantics but will motivate the fixed-point construction in LN 12.

The climax: we showed that the big-step rules map one-to-one to clauses in a JavaScript `evaluate()` function. **The inference rules are the specification; the interpreter is the implementation.** This is the first way to give Petal meaning: run it.

We saw tree-walking interpreters in the wild — from early Ruby to R to shell scripts — and noted that the pattern is the same: recursively walk the AST, applying the semantic rules at each node.

We saw **proof trees** as the formal mechanism for proving specific evaluation results — composing inference rules bottom-up, just like logical derivations in LN 2. We also explored what happens when proof trees fail: **stuck configurations** represent runtime errors, and **infinite proof trees** represent non-termination (a limitation unique to big-step semantics).

The key takeaway: **operational semantics is not just theory — it is engineering**. Every interpreter ever written follows this pattern. The gap between the mathematical specification (inference rules) and the running code (the `evaluate` function) is vanishingly small. When you understand the rules, you understand the interpreter, and vice versa.

Next lecture, we'll develop the second approach — **denotational semantics** — which gives meaning not by running programs, but by translating them into mathematical functions. And we'll see that this approach leads to a completely different implementation: a **transpiler**.
