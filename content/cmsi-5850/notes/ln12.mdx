## Recap

In LN 11, we formalized the first approach to semantics: **operational semantics — meaning by running**. We wrote inference rules (small-step and big-step) that define how Petal programs execute, then showed that each rule maps to a clause in a JavaScript `evaluate()` function. The inference rules were the specification; the interpreter was the implementation.

We extended Petal with `if-else` and `while`, and noticed something interesting about the while loop rule: it referred to itself. In operational semantics, that self-reference was perfectly fine — the rules define a relation, and the recursion unfolds naturally through repeated application. But we hinted that this would cause trouble in the next approach.

Today we develop the second approach: **denotational semantics — meaning by translating**. Instead of simulating how a program runs, we ask: *what mathematical function does this program compute?* And we'll see that this approach leads to a completely different implementation strategy: a **transpiler** that converts Petal programs into JavaScript source code.

## The Denotational Intuition

Operational semantics shows you *how* a computation unfolds — step by step, reduction by reduction. Denotational semantics takes a different stance: it doesn't care about the steps. It cares about the *result*.

Consider `3 + 5 * 2`. Operationally, this reduces in several steps: first `5 * 2 → 10`, then `3 + 10 → 13`. Denotationally, the expression simply **denotes** the number `13`. The steps don't matter; only the final mathematical object does.

> The word "denotation" comes from semiotics — the study of signs and symbols. In semiotics, the denotation of a word is the thing it refers to. The denotation of "cat" is the concept of a cat. Similarly, the denotation of `3 + 5` is the number 8 — not the process of adding, but the result itself.

<DefinitionBox term="Denotational Semantics">

**Denotational semantics** defines the meaning of a program by mapping it to a mathematical object — typically a function. The meaning is the mathematical object itself, not the process of computing it. Every program is viewed as a function from inputs to outputs.

</DefinitionBox>

The core idea: every program is a **function** — a mapping from inputs to outputs. Expressions are functions from environments to values. Statements are functions from states to states. Programs are functions from nothing to output. Sound familiar? These are the set-theoretic functions from LN 3 and the lambda abstractions from LN 4.

> Dana Scott and Christopher Strachey developed denotational semantics at Oxford in the late 1960s. Strachey provided the linguistic intuition; Scott provided the mathematical machinery (domain theory). Their collaboration was one of the most fruitful partnerships in the history of computer science — Strachey saw what languages *needed*, and Scott showed how to build it from mathematics.

## The Blueprint: From Tree Grammar to Semantic Functions

The tree grammar from LN 8 told us what AST shapes Petal programs can take. Now we need to assign mathematical meaning to each shape. Slonneger and Kurtz describe this as a three-step process: identify the **syntactic domains** (what goes in), define the **semantic domains** (what comes out), and write the **function signatures** connecting them.

### Recalling the Tree Grammar

In LN 8 we defined Petal's abstract syntax, and in LN 11 we extended it with control flow. Here is the current tree grammar:

<DisplayMath formula="e : \text{Exp} = \overline{n} \mid i \mid \text{-}e \mid e\text{+}e \mid e\text{-}e \mid e\text{*}e \mid e\text{/}e \mid e\text{\%}e \mid e\text{**}e" />

<DisplayMath formula="s : \text{Stmt} = i\text{=}e \mid \textbf{print}\; e \mid \textbf{if}\; e\; \textbf{then}\; s^+\; \textbf{else}\; s^+ \mid \textbf{while}\; e\; \textbf{do}\; s^+" />

<DisplayMath formula="p : \text{Program} = s^+" />

Each production names a syntactic category — a set of possible AST shapes. These categories are our **syntactic domains**: the inputs to the meaning functions we're about to define.

<DefinitionBox term="Syntactic Domain">

A **syntactic domain** is a set of abstract syntax trees belonging to a single grammatical category. For Petal, the syntactic domains are **Exp** (expressions), **Stmt** (statements), and **Program**. Each domain is defined by its tree grammar productions — they enumerate every possible shape an AST node of that category can take.

</DefinitionBox>

Think of the tree grammar as an API contract: every semantic approach must provide a complete implementation of this interface, handling every production, or programs will have undefined meaning. In LN 11, we wrote one inference rule per production. Here, we write one semantic equation per production. The tree grammar is the checklist.

### Semantic Domains

Now we define the mathematical spaces where meanings live — the **outputs** of our meaning functions.

<DefinitionBox term="Semantic Domain">

A **semantic domain** is a mathematical set that serves as the codomain of a meaning function. Unlike plain sets, domains carry additional structure — most importantly, they include a special element $\bot$ ("bottom") representing divergence or error.

</DefinitionBox>

For Petal:

<DefinitionBox term="Petal's Denotational Domains">

<DisplayMath formula="\text{Num} = \mathbb{Z} \cup \{\bot\}" />
<DisplayMath formula="\text{State} = \text{Identifier} \to \text{Num}" />
<DisplayMath formula="\text{Output} = \text{Num}^*" />

Where $\bot$ represents computations that produce no result — infinite loops, runtime errors, undefined behavior.

</DefinitionBox>

### The Bottom Element

<DefinitionBox term="Bottom ($\bot$)">

The **bottom element** $\bot$ ("bottom") is a special value that represents the absence of a result. It is the meaning of computations that diverge (loop forever), error (crash), or are undefined. Every domain includes $\bot$ as its least element.

</DefinitionBox>

The bottom element is crucial. Remember the $\Omega$ combinator from LN 4?

<DisplayMath formula="\Omega = (\lambda x.\, x\, x)(\lambda x.\, x\, x)" />

This expression reduces forever — it never reaches a value. In denotational semantics, its meaning is $\bot$. The symbol is the same ($\bot$) because the concept is the same: a computation with no result.

> You've already encountered $\bot$ without knowing it. In Haskell, the expression `undefined` has type `a` — it inhabits every type. Its denotation is $\bot$. Similarly, `error "message"` denotes $\bot$. In Rust, the `!` type (the "never" type) is the type of computations that never return — functions that panic or loop forever. These are all manifestations of the same mathematical concept.

**Domains vs. plain sets**: In LN 3, we worked with sets like $\mathbb{Z}$ and $\mathbb{R}$. Domains are these sets "lifted" to include $\bot$. We write $\text{Num}$ instead of $\mathbb{Z}$ to remind ourselves that the bottom element is always lurking. Every operation on domains must account for $\bot$: if any input is $\bot$, the output is $\bot$. This is called **strictness**.

<DefinitionBox term="Strictness">

A function $f$ is **strict** if $f(\bot) = \bot$. Most programming operations are strict: if an argument diverges, the whole expression diverges. A notable exception is short-circuit `&&` — if the left operand is false, the result is false regardless of whether the right operand diverges.

</DefinitionBox>

### From Domains to Functions

Now we can write the signatures that bridge syntax and semantics. Each syntactic domain gets its own semantic function:

<DefinitionBox term="Semantic Function">

A **semantic function** maps syntactic constructs to their meanings in the appropriate domain. Each syntactic category gets its own semantic function:

<DisplayMath formula="E : \text{Exp} \to \text{State} \to \text{Num}" />
<DisplayMath formula="S : \text{Stm} \to (\text{State} \times \text{Output}) \to (\text{State} \times \text{Output})" />
<DisplayMath formula="P : \text{Pro} \to \text{Output}" />

</DefinitionBox>

The double-bracket notation $E\llbracket e \rrbracket$ means "the meaning of expression $e$." We define these functions by structural induction on the AST — one equation per node type.

> The double-bracket notation $\llbracket \cdot \rrbracket$ (called "Oxford brackets" or "semantic brackets") was invented by Strachey. They serve as a visual reminder that we're crossing from the syntactic world (inside the brackets) to the semantic world (outside). Think of them as a function call: $E\llbracket e \rrbracket\, \sigma$ means "call the meaning function $E$ on expression $e$ with environment $\sigma$."

The rest of this lecture fills in these signatures — one equation per tree grammar production.

## Semantic Equations and Compositionality

### Expression Semantics

<DisplayMath formula="E\llbracket \overline{n} \rrbracket\, \sigma = \mathcal{N}\llbracket \overline{n} \rrbracket = n" />
<DisplayMath formula="E\llbracket i \rrbracket\, \sigma = \sigma(i)" />
<DisplayMath formula="E\llbracket e_1 + e_2 \rrbracket\, \sigma = E\llbracket e_1 \rrbracket\, \sigma + E\llbracket e_2 \rrbracket\, \sigma" />
<DisplayMath formula="E\llbracket e_1 * e_2 \rrbracket\, \sigma = E\llbracket e_1 \rrbracket\, \sigma \times E\llbracket e_2 \rrbracket\, \sigma" />

The <InlineMath formula="\overline{n}" /> on the left is a **numeral** — a syntactic object, a sequence of digit characters living in the AST. The <InlineMath formula="n" /> on the right is a **number** — a mathematical object living in the semantic domain $\text{Num}$. The valuation function <InlineMath formula="\mathcal{N}" /> bridges the two worlds. Every other expression equation builds on top of mathematical numbers; the <InlineMath formula="\mathcal{N}" /> bridge is the only place where syntax becomes semantics. Every subsequent <InlineMath formula="+" /> or <InlineMath formula="\times" /> on the right of an equation is the *mathematical* operation, not the syntactic operator that appears inside the brackets.

Each equation defines the meaning of one syntactic form. The equations are **compositional**: the meaning of a compound expression uses only the meanings of its subexpressions, never their internal structure.

We can also define additional operators:

<DisplayMath formula="E\llbracket e_1 - e_2 \rrbracket\, \sigma = E\llbracket e_1 \rrbracket\, \sigma - E\llbracket e_2 \rrbracket\, \sigma" />
<DisplayMath formula="E\llbracket e_1 / e_2 \rrbracket\, \sigma = \begin{cases} E\llbracket e_1 \rrbracket\, \sigma \div E\llbracket e_2 \rrbracket\, \sigma & \text{if } E\llbracket e_2 \rrbracket\, \sigma \neq 0 \\ \bot & \text{if } E\llbracket e_2 \rrbracket\, \sigma = 0 \end{cases}" />

Notice the division equation: when the divisor is zero, the result is $\bot$. This is how denotational semantics handles the division-by-zero error from LN 10 — the computation produces no value, represented by the bottom element. The error doesn't "crash" or "throw an exception"; it simply maps to $\bot$.

### Comparison Operators

For conditionals and loops, Petal needs comparison operators. These follow the same compositional template — evaluate both children, then return a numeric truth value:

<DisplayMath formula="E\llbracket e_1 > e_2 \rrbracket\, \sigma = \begin{cases} 1 & \text{if } E\llbracket e_1 \rrbracket\, \sigma > E\llbracket e_2 \rrbracket\, \sigma \\ 0 & \text{otherwise} \end{cases}" />
<DisplayMath formula="E\llbracket e_1 < e_2 \rrbracket\, \sigma = \begin{cases} 1 & \text{if } E\llbracket e_1 \rrbracket\, \sigma < E\llbracket e_2 \rrbracket\, \sigma \\ 0 & \text{otherwise} \end{cases}" />
<DisplayMath formula="E\llbracket e_1 \mathrel{=\!=} e_2 \rrbracket\, \sigma = \begin{cases} 1 & \text{if } E\llbracket e_1 \rrbracket\, \sigma = E\llbracket e_2 \rrbracket\, \sigma \\ 0 & \text{otherwise} \end{cases}" />

The remaining operators ($\leq$, $\geq$, $\neq$) follow the identical pattern; we omit them only because the equations are mechanical at this point. Comparisons produce integers (1 for true, 0 for false), matching the C convention we adopted in LN 11. Note also the careful symbol use: the <InlineMath formula="=\!=" /> *inside* the Oxford brackets is the syntactic operator (the AST node), while the <InlineMath formula="=" /> on the right of the case condition is the mathematical equality on $\text{Num}$.

### Statement Semantics

<DisplayMath formula="S\llbracket i\text{=}e \rrbracket\, (\sigma, o) = (\sigma[i \mapsto E\llbracket e \rrbracket\, \sigma],\; o)" />
<DisplayMath formula="S\llbracket \text{print } e \rrbracket\, (\sigma, o) = (\sigma,\; o \cdot E\llbracket e \rrbracket\, \sigma)" />
<DisplayMath formula="S\llbracket s_1;\, s_2 \rrbracket = S\llbracket s_2 \rrbracket \circ S\llbracket s_1 \rrbracket" />

Notice the sequencing equation: $S\llbracket s_1; s_2 \rrbracket = S\llbracket s_2 \rrbracket \circ S\llbracket s_1 \rrbracket$. The $\circ$ is **function composition** from LN 3. Sequential execution is just function composition! First apply $S\llbracket s_1 \rrbracket$ to the initial state, then apply $S\llbracket s_2 \rrbracket$ to the result.

> This is one of the most elegant insights of denotational semantics: imperative sequencing — the workhorse of procedural programming — is just function composition. The semicolon in `s1; s2` is mathematically identical to the $\circ$ operator on functions. This connects imperative and functional programming at the deepest level.

### Conditional Semantics

<DisplayMath formula="S\llbracket \textbf{if}\; e\; \textbf{then}\; s_1\; \textbf{else}\; s_2 \rrbracket\, (\sigma, o) = \begin{cases} S\llbracket s_1 \rrbracket\, (\sigma, o) & \text{if } E\llbracket e \rrbracket\, \sigma \neq 0 \\ S\llbracket s_2 \rrbracket\, (\sigma, o) & \text{if } E\llbracket e \rrbracket\, \sigma = 0 \end{cases}" />

The conditional is a function that selects between two other functions. No recursion, no fixed points — just a piecewise definition. This is why conditionals cause no theoretical difficulty in denotational semantics: they're perfectly compositional.

### Program Semantics

<DisplayMath formula="P\llbracket \text{program } s_1 \ldots s_n \rrbracket = \pi_2(S\llbracket s_n \rrbracket(\ldots S\llbracket s_1 \rrbracket(\sigma_0, \epsilon) \ldots))" />

where $\sigma_0$ is the initial (empty) state, $\epsilon$ is the empty output, and $\pi_2$ projects the output component.

### The Key Principle: Compositionality

Look at the equation for addition again:

<DisplayMath formula="E\llbracket e_1 + e_2 \rrbracket\, \sigma = E\llbracket e_1 \rrbracket\, \sigma + E\llbracket e_2 \rrbracket\, \sigma" />

<DefinitionBox term="Compositionality (Denotational)">

**Compositionality** is the requirement that the meaning of a compound expression is defined entirely in terms of the meanings of its subexpressions. Formally: $M\llbracket f(e_1, \ldots, e_n) \rrbracket = g(M\llbracket e_1 \rrbracket, \ldots, M\llbracket e_n \rrbracket)$ for some function $g$ that depends only on $f$. We never look "inside" the subexpressions — we only use their denotations.

</DefinitionBox>

This is not just an aesthetic choice — it's a **requirement**. Compositionality is what makes denotational semantics work: because ASTs are trees, and trees have parts, we can define meaning recursively from leaves to root.

This connects directly to LN 4. Church encodings *are* denotational semantics for arithmetic. When Church defined $\text{succ} = \lambda n.\, \lambda f.\, \lambda x.\, f\, (n\, f\, x)$, he was giving a compositional, mathematical meaning to the successor operation — the meaning of `succ(n)` is defined in terms of the meaning of `n`. The lambda calculus was the first denotational semantic framework.

> Compositionality has a powerful practical consequence: **modular reasoning**. If two subexpressions have the same denotation (they compute the same function), then replacing one with the other doesn't change the meaning of the program. This is the mathematical foundation of compiler optimization: the optimizer can replace `3 + 5` with `8` precisely because $E\llbracket 3 + 5 \rrbracket = E\llbracket 8 \rrbracket$ — they have the same denotation.

### Worked Example: Tracing the Semantic Functions

Let's trace a slightly larger program to see how the equations compose:

```
x = 3 + 5;
y = x * 2;
print y;
```

Starting with the initial state $(\sigma_0, \epsilon)$ where $\sigma_0$ is empty:

**Step 1:** Evaluate $S\llbracket x = \overline{3} + \overline{5} \rrbracket\, (\sigma_0, \epsilon)$

<DisplayMath formula="E\llbracket \overline{3} + \overline{5} \rrbracket\, \sigma_0 = E\llbracket \overline{3} \rrbracket\, \sigma_0 + E\llbracket \overline{5} \rrbracket\, \sigma_0 = 3 + 5 = 8" />
<DisplayMath formula="S\llbracket x = \overline{3} + \overline{5} \rrbracket\, (\sigma_0, \epsilon) = (\sigma_0[x \mapsto 8], \epsilon) = (\sigma_1, \epsilon)" />

**Step 2:** Evaluate $S\llbracket y = x * \overline{2} \rrbracket\, (\sigma_1, \epsilon)$

<DisplayMath formula="E\llbracket x * \overline{2} \rrbracket\, \sigma_1 = E\llbracket x \rrbracket\, \sigma_1 \times E\llbracket \overline{2} \rrbracket\, \sigma_1 = \sigma_1(x) \times 2 = 8 \times 2 = 16" />
<DisplayMath formula="S\llbracket y = x * \overline{2} \rrbracket\, (\sigma_1, \epsilon) = (\sigma_1[y \mapsto 16], \epsilon) = (\sigma_2, \epsilon)" />

**Step 3:** Evaluate $S\llbracket \text{print } y \rrbracket\, (\sigma_2, \epsilon)$

<DisplayMath formula="E\llbracket y \rrbracket\, \sigma_2 = \sigma_2(y) = 16" />
<DisplayMath formula="S\llbracket \text{print } y \rrbracket\, (\sigma_2, \epsilon) = (\sigma_2, \epsilon \cdot 16) = (\sigma_2, [16])" />

**Final result:** $P\llbracket \text{program} \rrbracket = \pi_2(\sigma_2, [16]) = [16]$

The program's denotation is the output `[16]`. Notice how each step is purely mathematical — we're computing with functions and sets, not simulating a machine. There are no "states" or "transitions"; there are only function applications.

## The Challenge: Loops and Fixed Points

Everything above works beautifully for straight-line code. But what about `while` loops? This is where denotational semantics faces its deepest challenge.

Consider the semantic equation for `while`:

<DisplayMath formula="S\llbracket \textbf{while}\; e\; \textbf{do}\; s \rrbracket\, (\sigma, o) = \begin{cases} (\sigma, o) & \text{if } E\llbracket e \rrbracket\, \sigma = 0 \\ S\llbracket \textbf{while}\; e\; \textbf{do}\; s \rrbracket\, (S\llbracket s \rrbracket\, (\sigma, o)) & \text{otherwise} \end{cases}" />

This equation defines <InlineMath formula="S\llbracket \textbf{while}\; e\; \textbf{do}\; s \rrbracket" /> **in terms of itself**. It's circular! In LN 11 we hinted that the while loop's self-referential rule would cause "trouble" in denotational semantics, and here it is: compositionality says the meaning of a compound construct must be defined from the meanings of its *parts*, and a `while` loop has no smaller `while` inside it to recurse into. Operational semantics escaped this trap because its inference rules were defining a **relation** — a set of triples — and "this triple is in the set if some other triple is in the set" is a perfectly good closure condition on a set. Denotational semantics cannot escape the same way. Our equation is supposed to *define a function* — a single mathematical object — and "<InlineMath formula="f" /> equals an expression containing <InlineMath formula="f" />" is satisfied by lots of different functions in general, so it does not, on its own, single out a unique meaning.

So we are stuck: we have written down an equation that *constrains* the meaning of `while`, but we have not yet written down enough machinery to say *which* function satisfying that constraint is "the" meaning. The next two subsections fix that. We will give the equation a name (a **functional** $F$), build successive **approximations** to its solution, and then introduce the mathematical structure (a partial order on functions, with a least element) that lets us pick out exactly one solution as the official meaning.

<DefinitionBox term="Fixed Point">

A **fixed point** of a function $F$ is a value $x$ such that $F(x) = x$ — applying the function to $x$ gives back $x$ itself. In denotational semantics, the meaning of a recursive construct (like `while`) is the **least fixed point** of a functional that captures one "unfolding" of the recursion.

</DefinitionBox>

The solution: **fixed points**. We rewrite the equation as:

<DisplayMath formula="S\llbracket \textbf{while}\; e\; \textbf{do}\; s \rrbracket = \text{fix}(F)" />

where $F$ is the functional:

<DisplayMath formula="F = \lambda w.\, \lambda (\sigma, o).\; \text{if } E\llbracket e \rrbracket\, \sigma = 0 \text{ then } (\sigma, o) \text{ else } w(S\llbracket s \rrbracket\, (\sigma, o))" />

The $\text{fix}$ operator finds a function $w$ such that $w = F(w)$ — a function that, when passed to $F$, produces itself back. This is the **Y combinator** from LN 4:

<DisplayMath formula="Y = \lambda f.\, (\lambda x.\, f\, (x\, x))(\lambda x.\, f\, (x\, x))" />

> The same mathematical trick that gave us recursion in the lambda calculus now gives us loops in denotational semantics. The Y combinator wasn't just an abstract curiosity — it's the mechanism that makes `while` loops meaningful. When we studied the Y combinator in LN 4, it seemed like a theoretical oddity. Here it reveals its true purpose: it's the bridge between compositionality and recursion.

### A Third Example: When the Result Depends on the Input

The structural pattern is identical to the subtraction loop (one new clause per step, the new clause boxed), but the *content* of each clause is what differs.

This is the deepest payoff of the denotational view. The loop's *meaning* is a closed-form mathematical function — Gauss's formula — and the iteration is just one operational way of computing it. Read the equation for $w_\infty$ above carefully: a smart compiler that recognises this denotation can replace the entire loop with the equivalent assignment

```

x = 0;

Production compilers genuinely do this kind of rewrite for definite loops with known closed forms (LLVM's `LoopIdiomRecognize`, GCC's `tree-scev`, and Polly all hunt for triangular and arithmetic-sum patterns). Where the operational view sees a loop running step by step, the denotational view sees Gauss's formula — and we'll see, in the *Denotational Equivalence and Optimization* section below, that this is exactly the principle that licences such rewrites in general.

### When You Don't Need Infinity

Reading the towers, it's tempting to think we have to "wait until infinity" to get a usable function. We don't. The infinite tower is what we need to give the loop a meaning *for every input at once*. For any *single* input, only a finite prefix of the tower matters — and we can often compute exactly which $w_k$ first catches it.

This is the difference between **definite** and **indefinite** iteration. A definite loop is one where the number of body iterations is determined by the input. Both of our examples are definite:

- For `while x > 0 { x = x - 1; }`, the body runs exactly $x$ times when $x \geq 0$. The smallest $w_k$ that handles input $x$ is <InlineMath formula="w_{x + 1}" />.
- For `while n > 1 { n = n / 2; }`, the body runs <InlineMath formula="\lfloor \log_2 n \rfloor" /> times when $n \geq 1$. The smallest $w_k$ that handles input $n$ is <InlineMath formula="w_{\lfloor \log_2 n \rfloor + 1}" />.
- For the accumulator `while x > 0 { s = s + x; x = x - 1; }`, the body runs exactly $x$ times — same closed form as the subtraction loop, <InlineMath formula="w_{x + 1}" />. The number of unfoldings is the same; the *content* of each unfolding is what changed, producing a non-trivial closed-form result.

Why bother with $w_\infty$ at all if every concrete input only needs a finite $w_k$? Because the *definition* of the loop's meaning has to work uniformly across every possible input — including inputs we haven't seen, or inputs whose required $k$ depends on runtime data. The fixed point packages "for every input, eventually" into a single mathematical object. For an *individual* input, that infinity is bookkeeping; for the *function as a whole*, it's the only honest answer.

> This is the same distinction we'll see in axiomatic semantics next lecture, where total correctness requires a **variant** — an explicit, finite measure of how many iterations remain. The variant is exactly the closed-form $k(x)$ from above; finding it is what proves termination. Domain theory and Hoare logic agree on the answer: definite loops have a meaning that requires only finite information per input.

> The approximation sequence reveals a beautiful connection: each $w_k$ is a **partial function** that "knows" how to handle inputs up to a certain complexity. As $k$ grows, the function handles more inputs. The fixed point is the function that handles *every* input — it's the "completed" version of the infinite process of approximation.

### Domain Theory (Intuition)

Why does the fixed-point construction work? Because domains aren't just sets — they have a **partial order** and they are **complete**.

<DefinitionBox term="Complete Partial Order (CPO)">

A **complete partial order** (CPO) is a set with a partial order <InlineMath formula="$\sqsubseteq$" /> (the "information ordering") such that every ascending chain <InlineMath formula="$x_0 \sqsubseteq x_1 \sqsubseteq x_2 \sqsubseteq \cdots$" /> has a least upper bound <InlineMath formula="$\bigsqcup_{i=0}^{\infty} x_i$" />. Domains are CPOs with a least element <InlineMath formula="$\bot$" />.

</DefinitionBox>

The information ordering says "$x \sqsubseteq y$" means "$y$ has at least as much information as $x$." The bottom element $\bot$ has no information at all — it represents total ignorance (divergence). A specific value like `42` has full information. In the approximation sequence:

<DisplayMath formula="\bot \sqsubseteq w_1 \sqsubseteq w_2 \sqsubseteq w_3 \sqsubseteq \cdots \sqsubseteq w_\infty" />

Each $w_k$ has more information than its predecessor (it handles more inputs). The fixed point $w_\infty$ is the limit — the function with all the information.

<DefinitionBox term="Kleene's Fixed-Point Theorem">

If $F$ is a continuous function on a CPO with bottom, then the least fixed point of $F$ exists and equals:

<DisplayMath formula="\text{fix}(F) = \bigsqcup_{i=0}^{\infty} F^i(\bot)" />

where $F^0(\bot) = \bot$, $F^1(\bot) = F(\bot)$, $F^2(\bot) = F(F(\bot))$, etc.

</DefinitionBox>

This structure guarantees that:

1. The approximation sequence $\bot, F(\bot), F(F(\bot)), \ldots$ is ascending
2. The sequence has a limit (the least fixed point)
3. The limit is the unique "best" solution to the equation $w = F(w)$

> We won't prove these theorems, but the intuition is essential: fixed points work because domains are the right mathematical habitat for recursive definitions. Plain sets aren't enough — you need the ordering and completeness that domains provide. This is why Dana Scott developed domain theory: to give denotational semantics a sound mathematical foundation for handling recursion.

### Why "Least" Fixed Point?

A functional $F$ might have multiple fixed points. Consider a function that maps every input to itself — *every* function is a fixed point of the identity functional. The **least** fixed point is the one with the least information — the one that diverges ($\bot$) on as many inputs as possible while still satisfying $w = F(w)$.

Why do we want the least? Because it corresponds to the **operational behavior**: a program computes exactly what its rules dictate, nothing more. If a while loop diverges for input $x = -1$, the operational semantics shows it running forever. The denotational semantics should give $\bot$ for that input — not some arbitrary value. The least fixed point guarantees this: it assigns $\bot$ to exactly those inputs where the loop doesn't terminate.

> This connects to a deep principle in logic: the **closed-world assumption**. If a fact cannot be derived from the rules, it is assumed false (or $\bot$). The least fixed point embodies this principle for programs: if a computation cannot be completed by the rules, its meaning is $\bot$.

### Recursive Functions (Not Just Loops)

The fixed-point construction isn't limited to `while` loops. Any recursive definition needs it. If Petal had recursive functions:

```
function factorial(n) {
  if n == 0 { return 1; }
  else { return n * factorial(n - 1); }
}
```

The denotation of `factorial` is $\text{fix}(F)$ where:

<DisplayMath formula="F = \lambda f.\, \lambda n.\, \text{if } n = 0 \text{ then } 1 \text{ else } n \times f(n-1)" />

The approximation sequence builds `factorial` incrementally: $w_0$ handles no inputs, $w_1$ handles $n = 0$, $w_2$ handles $n = 0, 1$, and so on. The fixed point handles all $n \geq 0$.

This is the same Y combinator from LN 4, applied to a different problem. The mathematical structure is identical — only the specific functional $F$ changes.

**What this whole arc earned us.** The same fixed-point machinery did two different jobs across these examples. For the subtraction and halving loops, the approximation sequence *confirmed* what your intuition probably already gave you: the final state collapses to a single value, edge cases aside. For the accumulator loop, your intuition almost certainly told you "it sums things into $s$" — but it did not hand you $s + \tfrac{x(x+1)}{2}$. For `factorial`, you may have already known the closed form $n!$, but the construction *built* it for you the same way it built Gauss's formula — incrementally, one input at a time, with no appeal to prior knowledge. That is the difference between a formalism that bookkeeps what you already know and a formalism that *discovers* what you don't. Denotational semantics does both, and the second is where it earns its keep — and where the next section's transpiler turns those discoveries into running code.

## From Equations to Code: The Transpiler

Here is today's payoff, paralleling LN 11's interpreter reveal. Every semantic equation maps to a clause in a `compile()` function that produces JavaScript source code:

<DefinitionBox term="Transpiler">

A **transpiler** (or **source-to-source compiler**) translates programs from one language to another at the same level of abstraction. Our Petal transpiler maps each AST node to the corresponding JavaScript construct, producing runnable JavaScript code. The transpiler IS the semantic equations turned into code.

</DefinitionBox>

```javascript
function compile(node) {
  switch (node.type) {
    case "Number":
      return `${node.value}`;

    case "Identifier":
      return node.name;

    case "BinaryOp":
      return `(${compile(node.left)} ${node.op} ${compile(node.right)})`;

    case "Assignment":
      return `let ${node.name} = ${compile(node.value)};`;

    case "Print":
      return `console.log(${compile(node.expression)});`;

    case "IfElse": {
      const thenBlock = node.then.map(compile).join("\n");
      const elseBlock = node.else.map(compile).join("\n");
      return `if (${compile(node.condition)}) {\n${thenBlock}\n} else {\n${elseBlock}\n}`;
    }

    case "While":
      return `while (${compile(node.condition)}) {\n${node.body.map(compile).join("\n")}\n}`;

    case "Program":
      return node.body.map(compile).join("\n");
  }
}
```

Compare each `case` to the corresponding semantic equation:

| Equation | Code |
|----------|------|
| $E\llbracket n \rrbracket\, \sigma = n$ | `case "Number"` — return the numeric value as a string |
| $E\llbracket i \rrbracket\, \sigma = \sigma(i)$ | `case "Identifier"` — return the variable name |
| $E\llbracket e_1 + e_2 \rrbracket = E\llbracket e_1 \rrbracket + E\llbracket e_2 \rrbracket$ | `case "BinaryOp"` — compile children, wrap with operator |
| $S\llbracket i = e \rrbracket$ | `case "Assignment"` — emit `let name = compile(value)` |

The correspondence is compositional, just like the equations. To compile a parent node, we compile its children and combine the results. The transpiler IS the semantic equations turned into code.

### The Same Pattern in Other Languages

```python
def compile(node):
    match node["type"]:
        case "Number":
            return str(node["value"])
        case "Identifier":
            return node["name"]
        case "BinaryOp":
            left = compile(node["left"])
            right = compile(node["right"])
            return f"({left} {node['op']} {right})"
        case "Assignment":
            return f"let {node['name']} = {compile(node['value'])};"
        case "Print":
            return f"console.log({compile(node['expression'])});"
```

```haskell
compile :: AST -> String
compile (Num n)          = show n
compile (Var x)          = x
compile (BinOp op l r)   = "(" ++ compile l ++ " " ++ [op] ++ " " ++ compile r ++ ")"
compile (Assign x e)     = "let " ++ x ++ " = " ++ compile e ++ ";"
compile (Print e)        = "console.log(" ++ compile e ++ ");"
```

> Notice how the transpiler is *structurally identical* to the interpreter from LN 11, but instead of computing values, it produces code strings. The interpreter `evaluate("Number", env) → 5` returns a number. The transpiler `compile("Number") → "5"` returns a string. Both follow the same recursive pattern because both are defined by structural induction on the same AST.

### Denotational Equivalence and Optimization

Two programs are **denotationally equivalent** if they have the same meaning function. This gives us a powerful tool for reasoning about programs:

<DisplayMath formula="E\llbracket (a + b) + c \rrbracket = E\llbracket a + (b + c) \rrbracket" />

because integer addition is associative. A compiler can freely rewrite `(a + b) + c` as `a + (b + c)` (which might be more efficient if `b + c` is a constant) because the denotation is the same.

Similarly:

<DisplayMath formula="E\llbracket x * 2 \rrbracket = E\llbracket x + x \rrbracket = E\llbracket x \ll 1 \rrbracket" />

Multiplying by 2, adding to itself, and left-shifting by 1 all have the same denotation. Compilers routinely replace multiplication by a power of 2 with a left shift — this optimization is justified by denotational equivalence.

#### A Formal Optimization Proof

Let's prove that **constant folding** — one of the most common compiler optimizations — is semantics-preserving. Consider replacing the expression `3 + 5` with `8` anywhere in a program.

**Claim:** $E\llbracket 3 + 5 \rrbracket\, \sigma = E\llbracket 8 \rrbracket\, \sigma$ for all states $\sigma$.

**Proof:**

<DisplayMath formula="E\llbracket 3 + 5 \rrbracket\, \sigma = E\llbracket 3 \rrbracket\, \sigma + E\llbracket 5 \rrbracket\, \sigma = 3 + 5 = 8 = E\llbracket 8 \rrbracket\, \sigma \quad \blacksquare" />

By compositionality, any program containing `3 + 5` as a subexpression can safely replace it with `8` without changing the program's denotation. This is constant folding — and we've just proven it correct using three lines of denotational reasoning.

More generally, any transformation that preserves the denotation is a valid optimization:

| Optimization | Denotational Justification |
|---|---|
| **Constant folding** | $E\llbracket 3 + 5 \rrbracket = E\llbracket 8 \rrbracket$ — compile-time evaluation |
| **Strength reduction** | $E\llbracket x * 2 \rrbracket = E\llbracket x \ll 1 \rrbracket$ — cheaper instruction, same function |
| **Dead code elimination** | If a branch is unreachable, removing it doesn't change the denotation |
| **Common subexpression elimination** | If $E\llbracket e \rrbracket$ appears twice and $e$ is pure, compute once and reuse |

> This is why denotational semantics is the preferred framework for compiler writers. Optimizations are **semantic preserving transformations** — they change the code but not the meaning. Compositionality guarantees that local changes (within a single expression) don't affect the meaning of the surrounding program. This is safety at the mathematical level.

### Extending Petal: Where the Math Lives

In LN 11, we noted that adding a new feature to a language activates the entire pipeline: concrete syntax, grammar, tree grammar, semantic rules. We also noted that if the feature relies on mathematics not yet in our toolkit, we must extend our foundations first. Denotational semantics makes this connection *literal* — the semantic equation for a new feature IS the mathematics of that feature.

Suppose we wanted to add a `integrate` keyword to Petal that computes definite integrals. The tree grammar gains a new expression production:

<DisplayMath formula="e : \text{Exp} = \ldots \mid \textbf{integrate}(e,\; i,\; e,\; e)" />

The four children are: the integrand expression, the variable of integration, the lower bound, and the upper bound. Now write the denotational equation:

<DisplayMath formula="E\llbracket \textbf{integrate}(e,\; i,\; a,\; b) \rrbracket\, \sigma = \int_{E\llbracket a \rrbracket\, \sigma}^{E\llbracket b \rrbracket\, \sigma} E\llbracket e \rrbracket\, \sigma[i \mapsto t]\; dt" />

The semantic equation *is literally an integral*. The Oxford brackets point directly to calculus. The meaning of `integrate(x*x, x, 0, 3)` is $\int_0^3 t^2\, dt = 9$ — a mathematical object, not an algorithm. If your language needs integration, your denotational semantics needs the mathematics of integration.

Contrast this with how operational semantics would handle the same feature. You'd write inference rules describing *how to compute* the integral — perhaps using Simpson's rule, the trapezoidal method, or symbolic integration steps. The operational rules specify an algorithm; the denotational equation specifies the mathematical result directly. One is a recipe; the other is the mathematical object itself.

The same pattern applies everywhere. A `sample` expression drawing from probability distributions would require the semantic domain to change from $\text{Num}$ to something like $\text{Distribution}(\text{Num})$ — the meaning of a stochastic program is a probability distribution, not a single number. A `derivative` keyword would map to $\frac{d}{dx}$. Every new mathematical capability in the language demands the same capability in the semantic domains. This is the deepest insight of denotational semantics: **the meaning of a program is exactly as rich as the mathematics it encodes**.

## Denotational Semantics in the Wild

LN 11 had its "Interpreters in the Wild" section: real systems whose entire architecture is recognisably an operational semantics walking an AST. Denotational semantics has the same kind of presence in production systems — but you have to know what you're looking at. The pattern to look for is *programs are functions, edits to programs are edits to those functions, evaluation is just function application*. Here are the four cleanest examples:

### Spreadsheets — the most successful denotational language ever shipped

If you have ever written `=A1 + B1 * 2` in a cell, you have written a denotational semantic equation. A spreadsheet is a denotational language to its bones:

- Each cell denotes a **value in a semantic domain** (a number, a string, a boolean, an error like `#DIV/0!` — exactly the spreadsheet's $\bot$).
- Each formula is a **compositional definition**: the meaning of `=A1 + B1` is *defined* in terms of the meanings of `A1` and `B1`. There is no "step" in spreadsheet evaluation, only the static dependency graph and the values it produces.
- The spreadsheet engine is a **transpiler-equivalent**: it does not "execute" your formula step by step the way an interpreter would; it *recomputes the function* whenever any input changes. Excel's recalculation engine is a fixed-point solver in disguise — it propagates updates until the entire grid is consistent with the equations you wrote.
- Recursion / circular references are handled by — surprise — **least fixed points**, with the option to enable iterative calculation (a Kleene approximation sequence in everything but name).

Spreadsheets are denotational semantics that won. Every accountant and operations analyst on Earth is, technically, writing denotational semantic equations every day. Most of them have never heard the word "denotational."

> The intellectual lineage is direct: in the late 1990s, Microsoft Research's *Lambda Excel* project (which evolved into LAMBDA, shipped in Excel 365 in 2021) was an explicit attempt to make Excel a Turing-complete *functional* language. Simon Peyton Jones — one of the architects of GHC and a long-time denotational semantics advocate — was on that team. Excel didn't accidentally become denotational; it got that way because that's what spreadsheets *are*.

### GHC Core — a real denotational compiler

The Glasgow Haskell Compiler (GHC) compiles Haskell down to a small intermediate language called **Core**, which is essentially typed lambda calculus. Every Core program *is* its own denotational semantic equation: a Haskell function literally translates to a lambda expression whose meaning, in the standard set-theoretic semantics for typed lambda calculus, is the function the original Haskell defined.

GHC's optimisation passes (inlining, common-subexpression elimination, strictness analysis, deforestation) are all rewrites *on Core*. They are sound *because* the rewriting is denotation-preserving — the Core program before and after the optimisation has the same mathematical meaning. This is the proof structure from the constant-folding example above, scaled up to a production compiler. When you read a GHC paper on a new optimisation pass, the proof of correctness is almost always "this rewrite preserves the denotation," verbatim.

### Proof assistants — Coq, Lean, Agda

Coq's `Gallina`, Lean 4's term language, and Agda are all dependently typed lambda calculi where:

- The **terms** are programs *and* proofs simultaneously (Curry–Howard).
- The **type** of a term is its specification — the denotation it must satisfy.
- The **kernel** decides equality of terms by reducing them to normal form: it computes denotations and compares them.

When you write a definition `fact : Nat -> Nat` in Lean 4, you are writing a denotational equation. When the kernel checks `fact 5 = 120`, it is verifying *denotational equivalence* — the two terms reduce to the same mathematical object. Theorems about your program are proven against its denotation, not against any execution trace.

### SQL (and relational query optimisers)

A SQL query like `SELECT name FROM users WHERE age > 30` *denotes* a set of rows — a mathematical object. The query planner is allowed to rewrite the query in arbitrarily wild ways (push down predicates, choose hash joins over merge joins, materialise sub-queries, exploit indices) because every legal rewrite is **denotation-preserving on the relational algebra**. The planner is, structurally, a constant-folder for relations. The reason `EXPLAIN` can show you fifteen different physical plans for the same SQL is that they all share a denotation.

> Across these four examples, the same picture keeps appearing: a program is a mathematical object, the runtime is "evaluate the object," and the optimiser's whole licence to reshape the program comes from preserving the object. The implementation differs wildly — Excel's recalc loop, GHC's Core simplifier, Lean's reducer, Postgres's planner — but the *theory* underneath is the lecture you're reading.

## Comparing Operational and Denotational

We now have two complete semantic frameworks for Petal, and two corresponding implementations:

| | Operational (LN 11) | Denotational (LN 12) |
|---|---|---|
| **Meaning** | How the program *runs* | What function it *computes* |
| **Notation** | Inference rules ($\Downarrow$, $\to$) | Semantic equations ($E\llbracket \cdot \rrbracket$) |
| **Loops** | Self-referential rules (fine) | Fixed points required |
| **Implementation** | Tree-walking interpreter | Transpiler to JavaScript |
| **Strengths** | Debugging, concurrency, non-termination | Optimization, equivalence proofs, composition |

For the same Petal program `x = 3 + 5; print x;`:

- The **interpreter** walks the AST, maintains an environment `{ x: 8 }`, and calls `console.log(8)` directly.
- The **transpiler** produces `let x = (3 + 5);\nconsole.log(x);` — a JavaScript program that, when run, produces the same output.

Both produce `8`. The **equivalence theorem** says they always agree on terminating programs: if the operational semantics says a program produces output $o$, then the denotational semantics maps that program to a function whose value is $o$, and vice versa.

### The Adequacy Theorem (and an Apparent Paradox)

<DefinitionBox term="Adequacy Theorem">

The **adequacy theorem** states that operational and denotational semantics agree: a program terminates with output $o$ under operational semantics if and only if its denotation maps the initial state to a result containing $o$. This theorem is the formal guarantee that interpreters and compilers produce the same results.

</DefinitionBox>

This statement is more surprising than it looks. We just spent a lecture insisting that operational and denotational semantics are *fundamentally different perspectives*: one is about how programs run, the other about what mathematical functions they compute. Loops in operational semantics are unproblematic; loops in denotational semantics demanded fixed points and a whole detour through domain theory. So how can the two perspectives agree on every input?

> **The apparent paradox.** If the two formalisms are *that* different — different machinery, different problems with `while`, different implementation strategies — how can they yield exactly the same answer on every program? Wouldn't *any* daylight between them produce a counterexample? It feels like the equivalence ought to be too good to be true.

The resolution is in two halves of the theorem, each with its own subtlety:

- **Soundness** (operational <InlineMath formula="\Rightarrow" /> denotational): if the operational rules say <InlineMath formula="\langle p, \sigma_0\rangle \Downarrow o" />, then <InlineMath formula="P\llbracket p \rrbracket = o" />. This direction is roughly an induction on the height of the operational derivation. The propagation/completion classification from LN 11 makes this clean: every propagation rule corresponds to an *equation that already holds* among the semantic functions, and every completion rule corresponds to a *base case* of one of those equations. The proof tree just *reuses* the equations.
- **Adequacy** (denotational <InlineMath formula="\Rightarrow" /> operational): if <InlineMath formula="P\llbracket p \rrbracket = o" /> for some non-<InlineMath formula="\bot" /> output <InlineMath formula="o" />, then the operational rules can derive <InlineMath formula="\langle p, \sigma_0\rangle \Downarrow o" />. This direction is harder, and it is the one that really uses the *least* fixed point. The denotational meaning of a `while` loop is built as the least upper bound <InlineMath formula="\bigsqcup_i w_i" /> of approximations; if <InlineMath formula="\bigsqcup_i w_i(\sigma) \neq \bot" />, then *some* finite <InlineMath formula="w_k(\sigma) \neq \bot" />, and that <InlineMath formula="k" /> tells us how many unfoldings of the operational `while` rule are needed.

What does the apparent disagreement on `while` *actually* mean, then? It is a disagreement about *what was hard to define*, not about *what gets computed*. The operational rules give a defining condition that any reasonable formalism would accept; the denotational equations give one that would be ambiguous without the least-fixed-point convention. The two formalisms reach the same destination by different routes, and that is the entire content of the theorem. Everything in this lecture — domains, <InlineMath formula="\bot" />, CPOs, Kleene's theorem — is the price of admission to a perspective in which programs are mathematical objects rather than processes; once paid, you get back exactly the same answer the operational semantics already gives.

> **Why care about a theorem that "just" reproduces what we already had?** Because the *new* perspective lets you do things the old one couldn't: you can compare programs by comparing functions ("are these two loops the same?"), prove optimizations sound by manipulating equations, and reason about *all* inputs at once instead of one execution at a time. The adequacy theorem is what licences trusting those denotational arguments back in the operational world your interpreter actually inhabits.

### The Implementation Duality

So far this course has paired each semantic approach with one implementation strategy: operational semantics with the tree-walking interpreter (LN 11), and now denotational semantics with the transpiler (LN 12). It is tempting to read those pairings as a strict one-to-one — "this kind of semantics produces that kind of implementation, full stop." That reading is a useful *pedagogical* framing, but it is a simplification.

The adequacy theorem proves that operational and denotational semantics are mathematically equivalent. This means either formalism can derive *either* implementation:

- **Denotational semantics as an interpreter**: Slonneger and Kurtz implement their denotational equations directly as a recursive interpreter in Prolog and Haskell. The semantic equations $E\llbracket e \rrbracket\, \sigma$ can be executed as a function `evaluate(e, sigma)` — instead of producing code strings, you compute values. The recursive structure is identical to our interpreter from LN 11.
- **Operational semantics as a compiler**: An operational semantics can be "partially evaluated" to produce a compiler. The inference rules define rewrite steps; a compiler pre-computes those steps at compile time, emitting target code that performs the remaining computation at runtime.

The interpreter/transpiler distinction is about *implementation strategy*, not about which formalism you started from. Both formalisms define the same mathematical object — the meaning of the program — and that meaning can be realized as either an interpreter or a compiler.

> This is why production compilers use *both* approaches: GCC uses operational reasoning for its intermediate representations (tracking execution state through SSA) and denotational reasoning for its optimization passes (proving that transformations preserve meaning). The two perspectives aren't competing — they're complementary tools in the same toolbox.

When does one approach shine over the other?

- **Operational** is better for describing **concurrency** (interleaving of steps), **debugging** (you can inspect intermediate states), and **non-termination** (small-step can show a program running forever).
- **Denotational** is better for **optimization** (two programs that denote the same function can be substituted for each other), **equivalence proofs** (comparing programs by comparing functions), and **modular reasoning** (compositionality means you can analyze parts independently).

> In practice, most language specifications use operational semantics (it's easier to read and implement), while most compiler optimizations rely on denotational reasoning (compositional equivalence enables transformations). The two approaches complement each other — understanding both makes you a more effective language designer and compiler writer.

### Interpreter vs. Transpiler: Side by Side

To make the structural similarity concrete, here are the two implementations side by side for the `BinaryOp` case:

```javascript
// INTERPRETER (LN 11): computes the VALUE
case "BinaryOp": {
  const left = evaluate(node.left, env);   // recursive call → number
  const right = evaluate(node.right, env); // recursive call → number
  return left + right;                      // returns a NUMBER
}

// TRANSPILER (LN 12): produces the CODE
case "BinaryOp": {
  const left = compile(node.left);          // recursive call → string
  const right = compile(node.right);        // recursive call → string
  return `(${left} + ${right})`;            // returns a STRING
}
```

The *structure* is identical: recurse into children, combine results. The *type* is different: the interpreter works with numbers, the transpiler works with strings. This isomorphism is a direct consequence of the adequacy theorem — the operational and denotational approaches are structurally the same computation viewed through different lenses.

### When Denotations Diverge

Not all language features have clean denotations. **Side effects** — I/O, mutation, exceptions, concurrency — complicate the denotational picture because they break compositionality. The denotation of `print(3) + print(5)` depends on the *order* in which the prints happen, which is an operational concern.

Haskell solves this with **monads** — a denotational framework for sequencing effects. Rust solves it with **ownership** — a type system that restricts when mutation can occur. Both approaches work to restore compositionality in the presence of effects, using different mechanisms from different parts of the theoretical landscape.

### Referential Transparency

<DefinitionBox term="Referential Transparency">

An expression is **referentially transparent** if it can be replaced by its value without changing the program's meaning. In denotational terms: if $E\llbracket e \rrbracket\, \sigma = v$, then replacing any occurrence of $e$ with $v$ in the program does not change its denotation.

</DefinitionBox>

Referential transparency is the practical consequence of compositionality. In a referentially transparent language (like pure Haskell), the compiler can freely:

- **Memoize**: compute `f(3)` once and reuse the result everywhere
- **Reorder**: evaluate `a + b` before or after `c + d` without affecting the result
- **Parallelize**: evaluate independent subexpressions on different CPU cores

In imperative languages (like C, Python, JavaScript), assignments break referential transparency. After `x = x + 1`, the expression `x` has a different value than before — you can't substitute the old value. This is why purely functional languages are easier to optimize and reason about, at the cost of requiring monads or similar mechanisms for side effects.

```haskell
-- Haskell: referentially transparent
let x = 3 + 5 in x * x
-- Can be rewritten as (3 + 5) * (3 + 5) or 8 * 8 — same meaning

-- Python: NOT referentially transparent
counter = 0
def inc():
    global counter
    counter += 1
    return counter
# inc() + inc() is NOT the same as 1 + 1 or 2 + 2 — order matters!
```

> Referential transparency connects denotational semantics back to mathematics itself. In algebra, $x + x$ always equals $2x$ — you can substitute freely. In imperative programming, this fails because variables change over time. Denotational semantics reveals this difference: expressions in a pure language denote *values*, while expressions in an imperative language denote *functions from states to values*. The extra argument (the state) is what makes substitution unsafe.

## Summary

Today we developed the second approach to semantics: **denotational semantics — meaning by translating**. We began with the **blueprint**: recalling the tree grammar and identifying the syntactic domains (Exp, Stmt, Program), semantic domains (Num, State, Output with $\bot$), and function signatures ($E$, $S$, $P$) that bridge them. The tree grammar serves as an API contract — each production demands one semantic equation. We carried the LN 11 numeral/number discipline forward: every semantic equation distinguishes the syntactic numeral $\overline{n}$ from the mathematical number $n$, with the valuation function $\mathcal{N}$ as the explicit bridge between worlds.

We wrote the equations and demonstrated **compositionality** — the defining principle — through the `DenotationalStepper`, watching semantic functions unfold through ASTs from leaves to root, and through the `SemanticEquationConstructor`, where every rewrite in a derivation chain has to be justified by a deliberately chosen equation from the palette. We saw that imperative sequencing is just function composition, and that Church encodings from LN 4 were denotational semantics all along.

We closed the LN 11 cliffhanger about why self-referential rules were fine for $\Downarrow$ (a relation) but problematic for $S\llbracket \cdot \rrbracket$ (a function). The fix is fixed points: `while` loops can't be defined compositionally without the $\text{fix}$ operator — the same Y combinator from LN 4 that gave us recursion. We watched the approximation sequence ($\bot$, $F(\bot)$, $F(F(\bot))$, ...) build the loop's meaning two ways at once: the `ApproximationTower` literally stacks each new equation atop the previous, with the freshly added clause boxed inside the formula, while the `ApproximationTable` records each $w_k$'s actual input-to-output behaviour, with cells either showing the direct one-step jump from input to result or a trailing $\bot$ when the function has no clause for that input. Three loops illustrated the range: the subtraction loop handles one new input per clause (collapsing every input to $0$); the halving loop handles an exponentially growing interval per clause (collapsing every input to $1$); and the accumulator loop adds one new input per clause but lands a *different* value each time, with $w_\infty$ turning out to be Gauss's formula $T_x = x(x+1)/2$. Same construction, three very different shapes of meaning. Crucially, every $w_k$ returns its answer in a *single* if/else-if cascade — the iteration lives in the *tower of approximations*, not in any single rung, which is precisely what allows a compiler to replace a loop with a closed-form assignment. We then made an honest distinction: the infinity in $w_\infty$ is for *generality*, not necessity. Definite loops have a closed-form $k(x)$, so any specific input is handled at a finite rung — a preview of next lecture's variant-based termination proofs. Kleene's theorem and CPO structure were what turned the ascending chain into a single, well-defined function.

We revealed the transpiler: each semantic equation maps to a clause in a `compile()` function that produces JavaScript. We proved that **denotational equivalence** justifies compiler optimizations like constant folding and strength reduction — safety at the mathematical level. We then surveyed denotational semantics **in the wild** — spreadsheets (the most successful denotational language ever shipped), GHC Core, proof assistants like Coq/Lean, and SQL query optimisers — all of which treat programs as mathematical objects whose meaning *is* their justification for any rewrite.

We explored what happens when Petal gains new features: an `integrate` keyword's semantic equation IS a literal integral, a `sample` keyword demands expanding semantic domains to probability distributions. This is the deepest insight — **denotational equations don't describe mathematics, they ARE mathematics**.

Finally, we compared the two frameworks and unpacked the **adequacy theorem** as its own subsection. The apparent paradox — "if the perspectives are this different, how can they ever agree?" — resolved into the soundness/adequacy split: every operational derivation reuses some sub-equation, and every non-$\bot$ denotation comes from a *finite* approximation $w_k$ that the operational rules can unfold $k$ times. The `InterpreterVsTranspilerSplit` made adequacy tactile: both lanes report the same $\sigma$ and $o$ at every step. The interpreter/transpiler mapping is a pedagogical convenience, not a rigid dichotomy — both formalisms can derive either implementation.

Next lecture, we complete the trio: **axiomatic semantics — meaning by proving**. We'll see that Hoare Logic lets us guarantee properties of programs without ever running them, and that this third perspective doesn't need an interpreter or a transpiler. It needs only logic.

**Key Definitions:**
- **Denotational Semantics**: meaning by translating — programs denote mathematical functions
- **Syntactic Domain**: a set of ASTs belonging to a grammatical category (Exp, Stmt, Program)
- **Semantic Domain**: a mathematical set with a partial order and a bottom element $\bot$
- **Bottom ($\bot$)**: represents divergence, error, or undefined — the denotational $\Omega$
- **Numeral vs. Number**: $\overline{n}$ is a syntactic digit string in the AST; $n$ is the mathematical number it denotes; $\mathcal{N}\llbracket \overline{n} \rrbracket = n$ is the explicit bridge
- **Semantic Function**: $E\llbracket e \rrbracket : \text{State} \to \text{Num}$ — maps syntax to meaning
- **Compositionality**: the meaning of a compound = function of meanings of its parts
- **Strictness**: $f(\bot) = \bot$ — divergent inputs produce divergent outputs
- **Transpiler**: source-to-source compiler; the implementation of semantic equations
- **Fixed Point**: a function $w$ such that $w = F(w)$ — the meaning of recursive constructs
- **Least Fixed Point**: the *unique* fixed point selected by Kleene's theorem; the convention that turns a recursive equation from a constraint into a definition
- **Approximation Sequence**: $\bot, F(\bot), F(F(\bot)), \ldots$ — each $w_k$ adds one more clause to the formula (the `ApproximationTower`) and gains a non-$\bot$ result on more inputs in the behaviour table (the `ApproximationTable`); each $w_k$ is itself a flat function that returns its answer in one if/else-if cascade
- **Definite Iteration**: a loop whose body executes a number of times determined by the input; the minimum $k$ such that $w_k$ catches a given input is computable in closed form, no fixed point required for that single input
- **CPO**: a partially ordered set where every ascending chain has a least upper bound
- **Kleene's Theorem**: the least fixed point exists and equals the limit of the approximation sequence
- **Adequacy Theorem**: operational and denotational semantics agree — split into *soundness* (operational $\Rightarrow$ denotational) and *adequacy* (denotational $\Rightarrow$ operational)
- **Denotational Equivalence**: two programs with the same meaning function — foundation of optimization
- **Referential Transparency**: an expression replaceable by its value without changing meaning
- **Function vs. Relation**: an inference rule may refer to its own relation (a set may be defined by closure); a denotational equation cannot refer to its own function without a least-fixed-point convention

- The tree grammar (LN 8) is the API contract: each production demands one semantic equation
