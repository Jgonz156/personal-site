### Properties of Complete Graphs

Complete graphs pack in every possible connection, which gives them a number of striking properties we can state using terms we've already built up:

**Regularity:** Every vertex in <InlineMath formula="K_n" /> has degree <InlineMath formula="n - 1" />. That makes <InlineMath formula="K_n" /> an <InlineMath formula="(n{-}1)" />-regular graph — every vertex is structurally identical.

**Complement:** The complement of <InlineMath formula="K_n" /> is the **empty graph** — the graph on <InlineMath formula="n" /> vertices with zero edges. Since <InlineMath formula="K_n" /> already has every possible edge, there's nothing left. Conversely, <InlineMath formula="G \cup \overline{G} = K_n" /> for any graph <InlineMath formula="G" /> on <InlineMath formula="n" /> vertices, as we saw in LN21.

**Resilience:** <InlineMath formula="K_n" /> has no cut vertices and no bridges. Removing any single vertex leaves <InlineMath formula="K_{n-1}" />, which is still complete — and therefore still connected. You'd need to remove <InlineMath formula="n - 1" /> vertices to disconnect it.

### Where Complete Graphs Appear

Complete graphs model "every pair interacts" scenarios, which arise more often than you might expect:

**Full-mesh networks:** In networking, a full-mesh topology connects every node to every other. This guarantees maximum fault tolerance — any single link failure still leaves all nodes reachable. But the cost is <InlineMath formula="\binom{n}{2}" /> connections, which grows quadratically. This is why full-mesh is practical only for small clusters; at data-center scale, engineers use cleverer topologies (like the hypercubes we'll see shortly).

**All-pairs testing:** In software QA, if you have <InlineMath formula="n" /> components that might interact, testing every pair requires <InlineMath formula="\binom{n}{2}" /> test cases — the edge set of <InlineMath formula="K_n" />. Python's `itertools.combinations(components, 2)` generates exactly these pairs.

**Round-robin tournaments:** A tournament where every team plays every other team is a complete graph with directed edges (the winner of each game gets an outgoing edge). With <InlineMath formula="n" /> teams, there are <InlineMath formula="\binom{n}{2}" /> games — and scheduling them efficiently is itself a combinatorics problem from LN17.

### The Odd-Cycle Characterization

**A graph is bipartite if and only if it contains no odd-length cycle.**

The intuition is elegant: in a bipartite graph, every walk alternates between the two parts <InlineMath formula="U" /> and <InlineMath formula="W" />. If you start in <InlineMath formula="U" />, after an even number of steps you're back in <InlineMath formula="U" />, and after an odd number you're in <InlineMath formula="W" />. An odd-length cycle would require you to return to your starting part after an odd number of steps — a contradiction.

This means <InlineMath formula="C_4" /> and <InlineMath formula="C_6" /> are bipartite (try splitting their vertices!), but <InlineMath formula="C_3" /> and <InlineMath formula="C_5" /> are not.

### Where Bipartite Graphs Appear

Bipartite graphs model any situation where two distinct groups interact, with connections only *between* groups, never *within*:

**Assignment and matching:** The classic use case. Students to projects, workers to tasks, applicants to positions, university courses to classrooms, organ donors to recipients. Whenever you're pairing items from two pools, you're working with a bipartite graph.

**Recommendation engines:** When Netflix suggests a movie or Amazon recommends a product, the underlying structure is a bipartite graph of users and items. Edges represent interactions (ratings, purchases). Finding patterns in this graph powers collaborative filtering — "users who liked X also liked Y."

**Compiler register allocation:** When a compiler assigns variables to CPU registers, it builds a bipartite graph mapping variables to available registers. Constraints (two variables alive at the same time can't share a register) restrict which edges are allowed.

**Biology:** Enzyme-substrate interaction networks are naturally bipartite — enzymes in one set, substrates in the other, edges representing catalytic relationships. Predator-prey food webs often have a bipartite core. In bioinformatics, sequence alignment can be modeled as bipartite matching between nucleotide positions.

In Python, a bipartite graph can be cleanly represented with two dictionaries tracking each partition:

```python
left = {"Alice", "Bob", "Carol"}
right = {"Project1", "Project2"}
edges = {("Alice", "Project1"), ("Bob", "Project1"), ("Carol", "Project2")}
```

### König's Theorem

One of the most elegant results about bipartite graphs is **König's theorem** (1931): in a bipartite graph, the size of the **maximum matching** (largest set of edges with no shared endpoints) equals the size of the **minimum vertex cover** (smallest set of vertices touching every edge). This duality fails for general graphs but always holds for bipartite ones — a beautiful structural consequence of having no odd cycles.

In Python, adjacency in <InlineMath formula="Q_n" /> has an elegant bitwise characterization:

```python
def adjacent_in_hypercube(a, b):
    return bin(a ^ b).count('1') == 1
```

Two vertices are adjacent iff their XOR has exactly one 1-bit — that's the **Hamming distance** being exactly 1.

### Hypercubes and Communication Hardware

Hypercubes aren't just mathematical curiosities — they shaped the history of parallel computing. In 1986, **Danny Hillis** built the **Connection Machine (CM-1)** at MIT, connecting 65,536 processors in a hypercube topology. Each processor could reach any other in at most <InlineMath formula="\log_2 n" /> hops — for 65,536 processors, that's just 16 hops. Intel's **iPSC** (Intel Personal SuperComputer) also used hypercube interconnects. Modern data center topologies like fat-trees and Clos networks evolved from these hypercube ideas, trading perfect symmetry for practical wiring constraints.

The hypercube also connects to **Gray codes**: a Hamiltonian path through <InlineMath formula="Q_n" /> visits every <InlineMath formula="n" />-bit string such that consecutive strings differ by exactly one bit. Gray codes are used in analog-to-digital converters and error correction, where changing multiple bits simultaneously could cause transient wrong values.

### Graph Theory in Biology

Graph theory reaches far beyond computer science. In **computational biology**, protein interaction networks map which proteins bind to one another in a cell — these networks exhibit "small-world" properties similar to hypercubes, where most proteins can reach any other through a short chain of interactions. Neural connectivity in the brain is modeled as a graph where neurons are vertices and synapses are edges. Even DNA assembly algorithms use graph traversal (Euler paths on de Bruijn graphs) to reconstruct genomes from short sequencing reads — a direct descendant of the bridge-walking we studied in LN20.

## Hamiltonian Graphs and the Petersen Graph

We now encounter a beautiful parallel to Euler's problem from LN20 — but with a twist that transforms the question from easy to extraordinarily hard.

**William Rowan Hamilton (1805–1865)** was an Irish mathematician and child prodigy who mastered multiple languages by age 13. He's famous for carving the quaternion equations into Brougham Bridge in Dublin in 1843. In 1857, he invented the **Icosian Game** — a commercial puzzle where players tried to find a path visiting every vertex of a dodecahedron exactly once. The irony: Hamilton's puzzle was a commercial failure, but the mathematical concept it inspired became one of the most studied problems in computer science.

<DefinitionBox term="Hamiltonian Path and Cycle">

A **Hamiltonian path** visits every **vertex** exactly once.

A **Hamiltonian cycle** visits every vertex exactly once and returns to the starting vertex.

A graph containing a Hamiltonian cycle is called **Hamiltonian**.

</DefinitionBox>

The parallel to Euler is striking: an Euler trail visits every *edge* once, a Hamiltonian path visits every *vertex* once. Same spirit, **vastly different difficulty**.

One immediate observation: every complete graph <InlineMath formula="K_n" /> with <InlineMath formula="n \geq 3" /> is Hamiltonian — since every vertex connects to every other, you can always find a cycle visiting them all. But as we'll see, having *fewer* edges makes the question dramatically harder.

Determining whether a graph is Hamiltonian is **NP-complete** — no known polynomial-time algorithm solves it. A small change in problem formulation ("every edge" vs "every vertex") transforms the problem from easy to intractable.

## Graph Theory Meets Optimization

Hamilton and Petersen showed us that some graph questions are *hard*. But difficulty doesn't mean we give up — it means we look for clever strategies. And here graph theory reveals one of its greatest strengths: it gives optimization problems a **visual structure**.

In our combinatorics unit (LN16–19), we counted arrangements, computed probabilities, and measured how fast possibilities grow. Optimization asks a related but different question: among all those arrangements, which is *best*? Which route is shortest? Which assignment costs the least? Which network is cheapest to build?

Graphs turn these abstract questions into something you can *see*. Routes become paths, assignments become matchings, networks become spanning trees. The vertices and edges aren't just notation — they're a map of the search space. And the tools we've built across this course — set operations, degree analysis, complement, connectivity — become the vocabulary for navigating that space.

The most famous optimization problem on graphs asks a deceptively simple question:

## The Traveling Salesman Problem

If every vertex must be visited, what's the **cheapest** way to do it?

<DefinitionBox term="Traveling Salesman Problem (TSP)">

Given a weighted complete graph on <InlineMath formula="n" /> cities, find the **Hamiltonian cycle of minimum total weight** — the cheapest round trip visiting every city exactly once.

</DefinitionBox>

Try mentally finding the cheapest route visiting all four cities. The brute-force approach: check all <InlineMath formula="n!" /> permutations. For 4 cities, that's 24 routes — manageable. But:

```python
from itertools import permutations
cities = list(range(20))
all_routes = list(permutations(cities))
# 2,432,902,008,176,640,000 routes — utterly infeasible
```

For 20 cities, brute force would outlast the sun. TSP is **NP-hard** — one of the most famous problems in all of computer science and mathematics.

The problem was formalized by **Karl Menger** in the 1930s in Vienna, but became famous when mathematicians at the **RAND Corporation** (Dantzig, Fulkerson, Johnson) solved a 49-city instance in 1954 — by hand, using clever cutting-plane methods. Today, the largest optimally solved instance has 85,900 cities (2006). The gap between "solved optimally" and "too large to solve" is one of the deepest open questions in all of mathematics — the **P vs NP** problem.

## Looking Ahead to LN23

We've now cataloged the key families of graphs and encountered one of the deepest divides in all of mathematics: problems that are easy to solve (Euler trails) versus problems that are impossibly hard (Hamiltonian cycles, TSP). But there's a family of graphs we haven't yet met — one so simple and so fundamental that it deserves its own lecture.

**Trees** are connected graphs with no cycles. Strip a connected graph down to its bare minimum — remove edges until one more removal would disconnect it — and what remains is a tree. If we can build a tree that **spans** every vertex of a graph using the cheapest edges, we get a **minimum spanning tree (MST)** — a problem with beautiful, efficient solutions that power real-world network design.

We'll also add **weights** to edges and ask a different optimization question: what's the **shortest path** between two vertices? **Dijkstra's algorithm** answers this with an elegant greedy strategy remarkably similar to how MSTs are built.

Finally, we'll peek beyond this course at three directions graph theory explores: **planarity** (can a graph be drawn without crossing edges?), **graph coloring** (remember the constraint puzzles?), and **network flows** (how much can flow through a graph?). These topics connect to circuit design, scheduling, and logistics — the foundations of entire subfields.

## Key Takeaways

<LectureNotes>

**Special Graph Families:**

- Cycle $C_n$: closed path of length $n$; bipartite iff $n$ is even
- Complete graph $K_n$: every pair connected, $\binom{n}{2}$ edges, <InlineMath formula="(n{-}1)" />-regular, complement is empty graph
- Bipartite: vertices split into two sets, edges only between sets; iff no odd-length cycle
- Complete bipartite <InlineMath formula="K_{m,n}" />: <InlineMath formula="m \cdot n" /> edges
- k-partite: $k$ independent sets, generalizes bipartite
- Wheel $W_n$: cycle plus hub, 3-connected (fault-tolerant)
- Hypercube $Q_n$: binary string vertices, adjacent iff Hamming distance = 1

**Key Properties:**

- König's theorem: in bipartite graphs, max matching = min vertex cover
- $K_n$ has no cut vertices or bridges — maximally resilient
- Constraint placement puzzles are constraint satisfaction problems (CSPs)

**Cycles and Hamiltonicity:**

- Hamiltonian path visits every vertex once; Hamiltonian cycle returns to start
- $K_n$ for $n \geq 3$ is always Hamiltonian
- Euler (every edge) → polynomial test. Hamilton (every vertex) → NP-complete
- Petersen graph: 3-regular but not Hamiltonian (classic counterexample)

**Optimization and TSP:**

- Graphs give optimization problems a visual structure
- TSP: find minimum-weight Hamiltonian cycle — NP-hard
- Brute force is $n!$ — factorial growth makes it infeasible
- One of the most studied problems in CS and optimization
