## Set Theory as an Alternative Foundation

Throughout this course, we used **type theory** as our foundation. Types gave us a clean way to classify objects, and the type signature <InlineMath formula="f : A \to B" /> gave us a precise way to talk about functions. But type theory is not the only option.

What if we could build *everything* from a single construction? What if numbers, tuples, functions — all of it — were secretly just **sets**?

> As your professor, I need to prove to you that this is possible. All I need to show is that numbers, collections, and functions can be constructed from just sets alone — then everything we have seen in this course falls out naturally via the same patterns between those constructions.

### Numbers from Sets — Von Neumann Ordinals

We need to build the natural numbers 0, 1, 2, 3, ... using nothing but sets. The construction, due to **John von Neumann**, is beautifully recursive:

<DefinitionBox term="Von Neumann Ordinals">

Define the natural numbers as:

- <InlineMath formula="0 = \emptyset = \{\}" />
- <InlineMath formula="1 = \{0\} = \{\emptyset\}" />
- <InlineMath formula="2 = \{0, 1\} = \{\emptyset, \{\emptyset\}\}" />
- <InlineMath formula="3 = \{0, 1, 2\} = \{\emptyset, \{\emptyset\}, \{\emptyset, \{\emptyset\}\}\}" />

In general: <InlineMath formula="n = \{0, 1, 2, \ldots, n-1\}" />

</DefinitionBox>

Each number *is* the set of all numbers before it. The number 3 literally contains 0, 1, and 2 as elements. Notice that <InlineMath formula="|n| = n" /> — the cardinality of the number equals the number itself. The **successor function** <InlineMath formula="S(n) = n \cup \{n\}" /> generates each number from the previous one. Step through the construction below:

<VonNeumannStepper />

Addition, multiplication, and all arithmetic can be built on top of the successor function through recursion.

### Tuples from Sets — Kuratowski Pairs

Tuples require ordered pairs — but sets are unordered. How do we recover ordering from sets?

<DefinitionBox term="Kuratowski Pair">

The **ordered pair** <InlineMath formula="(a, b)" /> is defined as:

<DisplayMath formula="(a, b) = \{\{a\}, \{a, b\}\}" />

</DefinitionBox>

Step through the construction below to see why this works — and why it recovers ordering from an unordered construction:

### Functions from Sets

With ordered pairs in hand, we can define functions as sets of pairs:

<DefinitionBox term="Function (Set-Theoretic)">

A **function** <InlineMath formula="f : A \to B" /> is a set of ordered pairs <InlineMath formula="f \subseteq A \times B" /> such that for every <InlineMath formula="a \in A" />, there exists exactly one <InlineMath formula="b \in B" /> with <InlineMath formula="(a, b) \in f" />.

</DefinitionBox>

But how far does this go? Let's take a familiar function — <InlineMath formula="\lambda x.\, x + 1" /> on <InlineMath formula="\{0, 1, 2\}" /> — and decompose it step by step into nothing but sets. First the function becomes pairs, then the pairs become Kuratowski sets, then the numbers become Von Neumann ordinals, until everything is just curly braces:

<SetDecompositionStepper />

> **Key Insight:** Everything is sets. Numbers are sets of sets. Tuples are sets of sets of sets. Functions are sets of tuples, which are sets of sets of sets of sets. It is sets all the way down.

## Why Set Theory?

Why would mathematicians choose to rebuild everything from sets when type theory works just fine?

| Criterion | Type Theory | Set Theory |
| :--- | :--- | :--- |
| **Primitives** | Multiple types, each with own rules | One construction: sets |
| **Consistency** | Checked per type | Checked once for the whole system |
| **Portability** | Tied to specific type definitions | Swap axioms, get different math |
| **CS Analogy** | Statically typed language (Haskell, Rust) | Dynamically typed language (Python) |

Three key advantages of set theory:

1. **Economy** — everything is a single construction. You only need to understand sets and set membership.
2. **Consistency** — because everything is sets, you only need to verify one system for contradictions, not many separate type systems.
3. **Portability** — the behavior of sets is determined by the axioms you choose. Change the axioms and you get a different mathematics. This makes set theory a *meta-framework* for building mathematical systems.

In Python terms: type theory is like a language with many built-in types (`int`, `str`, `list`, `dict`), each with its own operations. Set theory is like a language with only one type (`set`) — and everything else is built by nesting sets within sets. Both work. The question is which foundation you prefer.

## What Are Axioms?

We said that set theory defines all its behavior through axioms. But what is an axiom?

<DefinitionBox term="Axiom">

An **axiom** is a statement accepted as true without proof. Axioms serve as the starting rules of a mathematical system — the assumptions from which all theorems are derived.

</DefinitionBox>

You cannot prove axioms. That is the point. They are the foundation that everything else rests on. If you tried to prove an axiom, you would need to derive it from something more fundamental — but then *that* would be the axiom.

> **Side Note:** This might feel unsettling. Mathematics is supposed to be the most rigorous discipline — the one where everything is proven. And yet its very foundation consists of statements we simply *choose* to accept. We will return to this tension shortly.

### The ZFC Axioms

The most widely used axiom system for set theory is **ZFC** — Zermelo–Fraenkel set theory with the Axiom of Choice. Here are its nine axioms. Click any axiom to reveal its formal first-order logic definition:

<ZFCAxiomExplorer />

These nine statements — none of which are proven — generate all of mathematics. Every theorem in algebra, analysis, geometry, combinatorics, and graph theory can be derived from these axioms plus the rules of logic.

> **Historical Note:** **Ernst Zermelo (1871–1953)** published the first axiomatization of set theory in 1908 to address paradoxes like Russell's Paradox (the set of all sets that do not contain themselves — does it contain itself?). **Abraham Fraenkel (1891–1965)** refined the system in the 1920s. The Axiom of Choice, added by Zermelo, remains the most debated axiom — it asserts the existence of a selection function without constructing one, making it non-constructive.

## The "Faith" Conclusion

Here is the philosophical punchline of this lecture — and perhaps of the entire course.

Mathematics is built on axioms we *choose* to accept. We cannot prove the axioms themselves. All of mathematics ultimately rests on a decision to trust certain starting points.

This is not a weakness. It is what makes mathematics *flexible*.

<DisplayMath formula="\text{Axioms} \xrightarrow{\text{logical deduction}} \text{Theorems}" />

Different axiom choices produce different mathematical systems:

| Choice | Consequence |
| :--- | :--- |
| Accept the Axiom of Choice | Banach–Tarski paradox (decompose a sphere into pieces, reassemble into two spheres) |
| Reject the Axiom of Choice | Certain existence theorems fail; some constructions become impossible |
| Accept the Law of Excluded Middle | Classical logic; every statement is true or false |
| Reject the Law of Excluded Middle | Intuitionistic logic (LN1); constructive proofs required |

> **Key Insight:** In LN1, we started with intuitionism — a logic that *rejects* the law of excluded middle. Classical logic *accepts* it. Both are valid mathematical systems. The difference is a choice of axiom. You have been living in the space between these two foundations since day one of this course.

So all of mathematics is, at its deepest level, based on **faith** — a deliberate choice to accept certain starting points as given. This is not so different from how we build software: every program rests on assumptions about the operating system, the hardware, and the language runtime. We trust those foundations and build upward.

The axioms are our `import` statements for all of mathematics.

```python
from axioms import extensionality, empty_set, pairing, union
from axioms import power_set, infinity, replacement, foundation, choice
```

Everything else is derived.

- This flexibility is a feature: it allows mathematics to model any system

### Historical Context

- **Georg Cantor (1845–1918)** — Founded set theory to study the sizes of infinity; his work was initially controversial but became foundational
- **Ernst Zermelo (1871–1953)** — Published the first axiomatization of set theory (1908) to resolve paradoxes like Russell's Paradox

- **Kazimierz Kuratowski (1896–1980)** — Defined the ordered pair construction that bears his name

### Connection to Programming

- Python's `set` type implements many ZFC-like operations: `|` (union), `&` (intersection), `-` (difference), `<=` (subset)
- The Axiom of Extensionality is Python's `set.__eq__`: two sets are equal iff they have the same elements
- Type systems in languages like Haskell and Rust formalize the type theory approach; Python's dynamic typing is closer to set theory's "everything is one type"
- The `frozenset` type in Python corresponds to the set-theoretic requirement that set elements must be "well-founded" (no set contains itself)
