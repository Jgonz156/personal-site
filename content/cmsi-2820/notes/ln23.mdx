## Recap

Last lecture we cataloged the most important families of graphs — complete graphs, bipartite graphs, wheels, hypercubes, and the Petersen graph. We discovered the stark contrast between Euler's problem (every *edge* once — efficiently testable) and Hamilton's problem (every *vertex* once — NP-complete), and met the Traveling Salesman Problem.

## Trees

Strip a connected graph down to its bare minimum — remove edges until one more removal would disconnect it. What remains is a **tree**.

<DefinitionBox term="Tree">

A **tree** is a connected, acyclic graph. Equivalently, any two of these conditions imply the third:

1. <InlineMath formula="G" /> is connected
2. <InlineMath formula="G" /> is acyclic (no cycles)
3. <InlineMath formula="G" /> has exactly <InlineMath formula="n - 1" /> edges (where <InlineMath formula="n = |V|" />)

</DefinitionBox>

### Why These Three Conditions Travel Together

The definition says "any two imply the third," which is a strong claim. We'll see it through pictures: in each of the three pairings, the remaining condition is not a choice — it is *forced* by what the graph has to look like.

#### Pairing A: connected + acyclic forces <InlineMath formula="n - 1" /> edges

Start with <InlineMath formula="n" /> isolated vertices — that's <InlineMath formula="n" /> separate components. Now add edges one at a time, never closing a cycle (so we keep acyclic). Every such edge must **merge two different components into one**, because if an edge joined two vertices already in the same component, there'd already be a path between them — and the new edge would close a cycle.

    { id: "C", label: "C" }, { id: "D", label: "D" },
    { id: "E", label: "E" },
  ]}
  edges={[
    { from: "A", to: "B", color: "#16a34a", width: 3 },
    { from: "C", to: "D", color: "#16a34a", width: 3 },
  ]}
  interactive={true}
  height="240px"
/>

After 2 safe edges: 5 vertices, 3 components. Each edge ate exactly one component. So we're running a simple counter:

<DisplayMath formula="\text{components after } e \text{ edges} \;=\; n - e." />

To become connected we need 1 component, i.e. <InlineMath formula="n - e = 1" />, i.e. <InlineMath formula="e = n - 1" />. There is no wiggle room — connecting without cycling *costs exactly* <InlineMath formula="n - 1" /> edges.

#### Pairing B: connected + <InlineMath formula="n - 1" /> edges forces acyclic

A connected graph on <InlineMath formula="n" /> vertices needs **at least** <InlineMath formula="n - 1" /> edges (same counter argument — each edge removes at most one component). Suppose we have connectivity with exactly <InlineMath formula="n - 1" /> edges, and imagine a cycle was hiding inside.

The red edge C–A is part of the cycle A–B–C–A. Because cycles give *alternate routes*, we could delete C–A and the graph would stay connected — but now we'd have a connected graph on 5 vertices with only 3 edges. That's below the <InlineMath formula="n - 1" /> floor, which is impossible. So the cycle couldn't have been there in the first place. The edge budget <InlineMath formula="n - 1" /> is *so tight* that there's no room to waste an edge on redundancy.

#### Pairing C: acyclic + <InlineMath formula="n - 1" /> edges forces connected

Acyclic graphs are forests, and as we'll see in the next section a forest with <InlineMath formula="n" /> vertices and <InlineMath formula="k" /> components has exactly <InlineMath formula="n - k" /> edges.

A forest with just one tree is connected.

> **The takeaway:** these three conditions aren't a checklist of independent facts — they are three views of the same structure. Having any two is enough information to force the third, which is why computer scientists so often describe trees by whichever property is most convenient (a filesystem is "acyclic"; a spanning tree is "connected with <InlineMath formula="n - 1" /> edges"; a class hierarchy is "connected and acyclic").

### Rooting a Tree: From Graph Object to Hierarchy

Every tree we've drawn so far is "floating" — no vertex is privileged. But most real uses of trees in computer science pick a **root** and hang the tree from it. Once you root, the undirected tree becomes a hierarchy: each non-root vertex has exactly one **parent** (its neighbor on the path toward the root) and some number of **children** (its other neighbors).

Some vocabulary follows immediately:

- **Depth** of a vertex: length of the path from the root to it. The root has depth 0.
- **Leaf**: a vertex with no children (equivalently, degree 1 in the underlying tree — unless the tree is a single isolated root).
- **Subtree rooted at <InlineMath formula="v" />**: <InlineMath formula="v" /> together with all its descendants.
- **Height** of the tree: the maximum depth of any vertex.

Because there is **exactly one path** between any two vertices of a tree (a fact we prove visually next), the parent of every non-root vertex is unambiguous once a root is chosen. This is precisely why a filesystem path like `/home/alice/notes.txt` uniquely identifies a file: the filesystem is a tree rooted at `/`, and the path is just the unique sequence of edges from root to leaf.

> **Python note:** When you call `os.walk("/some/dir")`, Python is performing a depth-first traversal of a rooted tree. Picking the starting directory *is* picking the root.

### Unique Paths: The Defining Practical Property

Trees have one property that every application leans on:

> **Between any two vertices of a tree, there is exactly one path.**

We can see why visually. If two different paths existed between vertices <InlineMath formula="u" /> and <InlineMath formula="v" />, walking out along one and back along the other traces a closed loop — a cycle. But trees are acyclic, so two different paths can't both fit inside one.

Two different <InlineMath formula="u \to v" /> paths (blue and green) would instantly form the cycle <InlineMath formula="u \to x \to v \to y \to u" />. So this picture cannot be a tree — any graph that *is* a tree has, for each pair of vertices, just one path between them.

This "unique-path" property is what powers: URL routing (each route is one path in the routing tree), DOM traversal in HTML (each element has one path from `<html>`), git ancestry in a linear history, and lookups in every balanced search tree.

> **Every tree is bipartite.**

When we reach graph coloring later in this lecture, this lets us immediately conclude <InlineMath formula="\chi(T) = 2" /> for any tree <InlineMath formula="T" /> with at least one edge — no case analysis needed.

### Counting with Degrees in a Tree

The handshake identity (from LN21) says that in any graph, the sum of degrees equals twice the number of edges. For a tree we know <InlineMath formula="|E| = n - 1" />, so:

<DisplayMath formula="\sum_{v \in V} \deg(v) = 2(n - 1)." />

One elegant consequence shows up in **proper binary trees** — rooted trees where every non-leaf has *exactly* two children. Let <InlineMath formula="n_0" /> count the leaves and <InlineMath formula="n_2" /> count the internal (two-child) nodes. Then:

- <InlineMath formula="n = n_0 + n_2" /> (every node is either a leaf or a two-child internal node)
- <InlineMath formula="|E| = n - 1" /> (it's a tree)
- <InlineMath formula="|E| = 2 \, n_2" /> (every edge is a parent-to-child edge, and each internal node contributes exactly two such edges)

Setting the two expressions for <InlineMath formula="|E|" /> equal: <InlineMath formula="2 \, n_2 = n_0 + n_2 - 1" />, which rearranges to:

<DisplayMath formula="n_0 = n_2 + 1." />

**In a proper binary tree, the number of leaves is always exactly one more than the number of internal nodes.**

A surprising amount of predictable structure falls out of nothing more than edge counting — and this identity is why balanced binary heaps and BSTs have such regular shapes.

### Trees Everywhere in CS

Trees are arguably the most important structure in all of computer science, and you've been using them without knowing it:

- **File systems**: `os.walk()` traverses the directory tree. Every path from root `/` to a file is unique — that's because it's a tree
- **Abstract Syntax Trees**: `import ast; ast.parse("x + 1")` produces a tree. Python parses every program into a tree before executing it (callback to LN12–14 — lambda calculus expressions are parsed into trees too)
- **Class hierarchies**: Python's MRO (Method Resolution Order) linearizes the inheritance tree. `ClassName.__mro__` shows the path
- **JSON/nested dicts**: `json.loads()` produces nested dicts — a tree structure where every key path is unique

## Two Intuitions for Building Trees

Trees sit at exactly <InlineMath formula="n - 1" /> edges — the minimum for connectivity. We can approach this number from two directions:

**Cutting down** (top-down): Start from <InlineMath formula="K_n" /> — the maximally connected graph. Remove edges while maintaining connectivity until you reach <InlineMath formula="n - 1" /> edges. You must have a tree.

**Building up** (bottom-up): Start from <InlineMath formula="n" /> isolated vertices (a forest of <InlineMath formula="n" /> components). Add edges without creating cycles until you have exactly one component. You must have a tree.

## Spanning Trees

<DefinitionBox term="Spanning Tree">

A **spanning tree** of a connected graph <InlineMath formula="G" /> is a subgraph that:
1. Is a tree (connected and acyclic)
2. Includes **all vertices** of <InlineMath formula="G" />

Every connected graph has at least one spanning tree (by the "cutting down" process above — keep removing cycle edges until none remain).

</DefinitionBox>

So far, all edges have been equal. What if some connections are more expensive, longer, or faster than others?

</DefinitionBox>

### Applications and Algorithms

MSTs arise naturally across many fields:
- **Network design**: minimum cable to connect buildings
- **Circuit design**: minimum wire to connect components
- **Clustering**: group data points by removing the longest MST edges
- **Approximation algorithms**: MSTs provide near-optimal solutions for TSP

Three classical algorithms find MSTs efficiently:

| Algorithm | Year | Strategy | Intuition |
| :--- | :---: | :--- | :--- |
| Borůvka's | 1926 | Every component picks its cheapest outgoing edge simultaneously | Parallel "building up" |
| Kruskal's | 1956 | Sort all edges by weight, add cheapest that doesn't create a cycle | Sequential "building up" |
| Prim's | 1957 | Grow outward from one vertex, always adding the cheapest edge to a new vertex | "Growing outward" |

Notice the connection to our two tree-building intuitions: Kruskal's mirrors "building up" (adding edges without cycles), while Prim's mirrors "growing outward."

You will implement all three algorithms in Python in the HW6 programming assignment (`prim.py`, `kruskal.py`, `boruvka.py`), which provides a guided, hands-on treatment.

> **Historical Note:** **Otakar Borůvka (1899–1995)** developed the first MST algorithm not as an abstract exercise, but to solve a real engineering problem: **electrifying Moravia**. The regional power company needed to connect cities with minimum total wire. His solution — let every city independently pick its cheapest outgoing connection, merge the resulting components, and repeat — predates Kruskal's and Prim's by three decades. He lived to 96 and saw his algorithm rediscovered for parallel computing in the 1990s.

## Shortest Paths

The second optimization question on weighted graphs: given a source vertex, find the minimum-weight path to every other vertex.

<DefinitionBox term="Shortest Path">

The **shortest path** from vertex <InlineMath formula="s" /> to vertex <InlineMath formula="t" /> in a weighted graph is the path from <InlineMath formula="s" /> to <InlineMath formula="t" /> with minimum total edge weight.

</DefinitionBox>

> **Key Insight:** MST and shortest path answer *different* questions: MST connects everyone as cheaply as possible; shortest path finds the cheapest route between two specific nodes. The MST does **not** guarantee shortest paths between specific pairs — it minimizes *total* cost, not *pairwise* cost.

To see the difference:

The **shortest path** from A to E goes A → C → D → E with total weight 2 + 3 + 7 = **12** (blue edges). But the MST path from A to E is the same in this case. In general, the MST does **not** guarantee shortest paths between specific pairs — it minimizes *total* cost, not *pairwise* cost.

> **Historical Note:** **Edsger Dijkstra (1930–2002)** designed his shortest path algorithm in about 20 minutes while sitting at a café in Amsterdam in 1956. His original motivation: "What is the shortest way to travel from Rotterdam to Groningen?" The algorithm greedily expands a frontier from the source, always picking the closest unvisited vertex — similar to Prim's (both are greedy, both grow outward from a single node). Dijkstra went on to win the Turing Award in 1972, coining "structured programming" and arguing passionately against `goto` statements.

In Python, both Prim's and Dijkstra's rely on efficiently finding the minimum — Python's `heapq` module implements a binary heap, which is itself a tree data structure:

```python
import heapq
frontier = [(2, "C"), (4, "B")]  # (weight, node)
heapq.heappop(frontier)          # (2, "C") — always returns minimum
```

Applications of shortest paths: GPS navigation, network routing (OSPF — the protocol that routes packets across the internet), and game pathfinding.

## Where Graph Theory Goes from Here

We have covered the fundamentals of graph theory in four lectures. But this is just the beginning — graph theory is a living, expansive field. Here are three directions that connect directly to what we've learned.

### Planarity

Can a graph be drawn in the plane without any edge crossings? A graph with this property is called **planar**. This is relevant to circuit board design — components on a single-layer board can only be connected without crossing wires if the circuit graph is planar.

<GraphVisualization
  nodes={[

    { id: "C", label: "C" }, { id: "D", label: "D" },

  edges={[
    { from: "A", to: "B" }, { from: "B", to: "C" },
    { from: "C", to: "D" }, { from: "D", to: "A" },
    { from: "A", to: "C" },
  ]}
  interactive={true}
  height="260px"
/>

This graph is planar — it can be drawn without crossings. But try drawing <InlineMath formula="K_5" /> or <InlineMath formula="K_{3,3}" /> (from LN22) without crossings — you can't. **Kuratowski's theorem** says a graph is planar if and only if it contains no "copy" of <InlineMath formula="K_5" /> or <InlineMath formula="K_{3,3}" /> (technically, no subdivision of either as a subgraph).

For planar graphs, Euler discovered another elegant formula: <InlineMath formula="V - E + F = 2" />, where <InlineMath formula="F" /> is the number of faces (regions) in the planar drawing. This invariant holds for every planar graph.

### Graph Coloring

Assign colors to vertices so that **no two adjacent vertices share a color**. The minimum number of colors needed is the **chromatic number** <InlineMath formula="\chi(G)" />.

<GraphVisualization
  nodes={[
    { id: "A", label: "A", color: "#fca5a5" },
    { id: "B", label: "B", color: "#93c5fd" },
    { id: "C", label: "C", color: "#fca5a5" },
    { id: "D", label: "D", color: "#93c5fd" },

  ]}
  edges={[
    { from: "A", to: "B" }, { from: "B", to: "C" },
    { from: "C", to: "D" }, { from: "D", to: "E" },
    { from: "E", to: "A" },
  ]}
  interactive={true}
  height="280px"
/>

Notice how we alternate red and blue around the cycle — then hit E, which is adjacent to both a blue (D) and a red (A), forcing a third color. That "parity break" is exactly why odd cycles resist 2-coloring. This <InlineMath formula="C_5" /> requires 3 colors (<InlineMath formula="\chi(C_5) = 3" />). The constraint placement puzzles from LN22 were secretly graph coloring problems — you were assigning "colors" (letters) to vertices subject to adjacency constraints. You solved a coloring problem without knowing it!

#### Chromatic Numbers of Familiar Families

Here's where the vocabulary from LN22 pays off. We can immediately determine the chromatic number of every graph family we studied:

- <InlineMath formula="\chi(K_n) = n" />: In a complete graph, every vertex is adjacent to every other, so every vertex needs its own color.
- <InlineMath formula="\chi(C_n) = 2" /> if <InlineMath formula="n" /> is even, <InlineMath formula="\chi(C_n) = 3" /> if <InlineMath formula="n" /> is odd: Even cycles can alternate two colors; odd cycles need a third.
- <InlineMath formula="\chi(G) = 2" /> for any bipartite graph <InlineMath formula="G" />: The two partitions *are* the two color classes. This is precisely why bipartiteness and 2-colorability are equivalent.
- <InlineMath formula="\chi(T) = 2" /> for any tree <InlineMath formula="T" /> (with at least one edge): Trees are bipartite (no cycles at all, so certainly no *odd* cycles).

#### The k-Partite Connection

In LN22, we noted that k-partite graphs partition vertices into independent sets — groups with no internal edges. Graph coloring reveals why that matters: **a graph is k-colorable if and only if it is a subgraph of a complete k-partite graph**. Each color class is an independent set, and the partition into color classes is precisely a k-partition. The concept of k-partite graphs and graph coloring are two sides of the same coin.

#### Ramsey Theory: Coloring Complete Graphs

Here's a surprising result about coloring. Take <InlineMath formula="K_6" /> and color every *edge* either red or blue — any way you like. No matter how you choose, **a monochromatic triangle must exist**: three vertices whose connecting edges are all the same color. This is the simplest case of **Ramsey theory**, which studies what structures inevitably emerge as graphs grow. The result connects complete graphs (LN22) to coloring in an unexpected way — even at the edge level, structure is inescapable.

#### The Four-Color Theorem

> **Historical Note:** The **four-color theorem** — one of the most famous results in mathematics — states that every planar graph can be colored with at most 4 colors. It was the first major theorem proved with the aid of a computer (1976), when **Kenneth Appel** and **Wolfgang Haken** reduced it to checking 1,936 configurations by machine.

Applications: exam scheduling (no student has two exams at the same time), register allocation in compilers, and the classic problem of map coloring.

### Network Flows

Model a graph as a network with **capacities** on edges. How much "stuff" (data, water, traffic) can flow from a source to a sink?

## Key Takeaways

<LectureNotes>

**Trees and Forests:**

- Tree: connected + acyclic + $n-1$ edges (any two imply the third)
- Forest: acyclic graph; each component is a tree; $n$ vertices, $k$ components → $n-k$ edges
- Trees are fragile: removing any edge disconnects, adding any edge creates a cycle
- Trees are everywhere in CS: file systems, ASTs, class hierarchies, JSON, heaps

**Two Intuitions for Trees:**

- **Cutting down**: remove edges from $K_n$ while staying connected → tree
- **Building up**: add edges to isolated vertices without creating cycles → tree
- These mirror Kruskal's (building up) and Prim's (growing outward) MST algorithms

**Spanning Trees and MSTs:**

- Spanning tree: subgraph that is a tree and includes all vertices
- MST: spanning tree with minimum total edge weight
- Three classical algorithms: Borůvka (1926), Kruskal (1956), Prim (1957)
- Applications: network design, circuit layout, clustering, TSP approximation

**Weighted Graphs and Shortest Paths:**

- Weighted graph: <InlineMath formula="(V, E, w)" /> with <InlineMath formula="w: E \to \mathbb{R}" />
- MST ≠ shortest paths — different optimization objectives
- Dijkstra's algorithm: greedy expansion from source, similar to Prim's
- `heapq` in Python implements the priority queue both algorithms need

**Where Graph Theory Goes Next:**

- Planarity: <InlineMath formula="K_5" /> and <InlineMath formula="K_{3,3}" /> are not planar; Euler's formula <InlineMath formula="V - E + F = 2" />
- Coloring: chromatic number <InlineMath formula="\chi(G)" />; <InlineMath formula="\chi(K_n) = n" />, <InlineMath formula="\chi(\text{bipartite}) = 2" />, <InlineMath formula="\chi(\text{tree}) = 2" />; four-color theorem for planar graphs
- k-colorable iff subgraph of complete k-partite — LN22 vocabulary meets coloring
- Ramsey theory: monochromatic triangles inevitable in 2-edge-colored $K_6$
- Network flows: max-flow/min-cut theorem
