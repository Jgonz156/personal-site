## Connectedness

Here is a simple but fundamental question: given two vertices in a graph, can you get from one to the other by following edges?

  interactive={true}
  height="300px"

In this graph, you can walk from A to B to C and back — but there is **no path** from A to D. The graph is split into two unreachable clusters.

<DefinitionBox term="Connected Graph">

A graph <InlineMath formula="G = (V, E)" /> is **connected** if for every pair of vertices <InlineMath formula="u, v \in V" />, there exists a path from <InlineMath formula="u" /> to <InlineMath formula="v" />.

A graph that is not connected is **disconnected**.

</DefinitionBox>

In Python, we can check connectivity with a BFS using only data structures from LN9 — a `set` for visited nodes and a list for the queue:

```python
def is_connected(graph):
    start = next(iter(graph))
    visited = {start}
    queue = [start]
    while queue:
        node = queue.pop(0)
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append(neighbor)
    return len(visited) == len(graph)
```

If we can reach every vertex from any starting vertex, the graph is connected.

## Graph Unions

There is an elegant set-theoretic way to express disconnectedness. First, we need to define the **union** of two graphs:

<DefinitionBox term="Graph Union">

Given graphs <InlineMath formula="G_1 = (V_1, E_1)" /> and <InlineMath formula="G_2 = (V_2, E_2)" />, their **union** is:

<DisplayMath formula="G_1 \cup G_2 = (V_1 \cup V_2,\; E_1 \cup E_2)" />

</DefinitionBox>

This is a direct application of set union (LN10) on both components of the graph tuple.

## Components

Every graph — connected or not — can be uniquely decomposed into its maximal connected pieces:

<DefinitionBox term="Connected Component">

A **connected component** of a graph <InlineMath formula="G" /> is a maximal connected subgraph — a connected piece that cannot be made larger by adding more vertices from <InlineMath formula="G" />.

Every graph decomposes uniquely into its connected components.

</DefinitionBox>

## Cut Vertices and Bridges

We know what connected means. Now: how *fragile* is that connectivity? What if removing a single vertex or edge breaks the graph apart?

<DefinitionBox term="Cut Vertex and Bridge">

A **cut vertex** (articulation point) of a connected graph is a vertex whose removal **disconnects** the graph.

A **bridge** is an edge whose removal **disconnects** the graph.

</DefinitionBox>

Using the lambda calculus as a tool (LN19), we can express these tests formally — building the modified graph with set difference (LN10), then checking connectivity:

<DisplayMath formula="\text{is_cut_vertex} = \lambda G.\; \lambda v.\; \text{connected}(G) \;\wedge\; \neg\,\text{connected}\!\big((V \setminus \{v\},\; \{e \in E \mid v \notin e\})\big)" />

<DisplayMath formula="\text{is_bridge} = \lambda G.\; \lambda e.\; \text{connected}(G) \;\wedge\; \neg\,\text{connected}\!\big((V,\; E \setminus \{e\})\big)" />

The cut vertex lambda removes <InlineMath formula="v" /> from <InlineMath formula="V" /> and filters out all incident edges; the bridge lambda removes just one edge from <InlineMath formula="E" />. Both then check whether the resulting graph is still connected.

### Measuring Connectivity: Bisection Width

Cut vertices and bridges ask: "Can we break this graph by removing *one* thing?" A stronger question is: how many edges must we remove to split the graph roughly in half?

<DefinitionBox term="Bisection Width">

The **bisection width** of a graph <InlineMath formula="G" /> with <InlineMath formula="n" /> vertices is the minimum number of edges whose removal splits <InlineMath formula="G" /> into two parts of sizes <InlineMath formula="\lfloor n/2 \rfloor" /> and <InlineMath formula="\lceil n/2 \rceil" />.

</DefinitionBox>

Here is a graph where the minimum bisection cuts exactly 2 edges — the two cross-connections (highlighted in red) between the left and right clusters:

The bisection <InlineMath formula="\{J, K, L\} \mid \{M, N, O\}" /> requires cutting exactly 2 edges, so this graph has bisection width 2. No bridge exists (removing just one red edge still leaves the graph connected via the other), but the graph can still be cleanly halved.

Bisection width matters for **network design**. In supercomputer interconnects, a high bisection width means data can flow between any two halves of the machine with high bandwidth — there are many independent paths. A hub-and-spoke network (like a star graph, where one central node connects to all others) has bisection width 1 — a single edge removal halves the network. Data center architects actively optimize for high bisection width to avoid bottlenecks.

## Subgraphs

<DefinitionBox term="Subgraph and Induced Subgraph">

<InlineMath formula="H = (V_H, E_H)" /> is a **subgraph** of <InlineMath formula="G = (V, E)" /> if <InlineMath formula="V_H \subseteq V" /> and <InlineMath formula="E_H \subseteq E" />.

The **induced subgraph** <InlineMath formula="G[S]" /> for a vertex subset <InlineMath formula="S \subseteq V" /> includes all edges from <InlineMath formula="G" /> that have *both* endpoints in <InlineMath formula="S" />.

</DefinitionBox>

These definitions translate directly into lambdas using the set operations from LN10. The subgraph test checks subset inclusion on both components of the tuple; the induced subgraph lambda constructs a new graph, keeping only edges whose both endpoints are in <InlineMath formula="S" />:

<DisplayMath formula="\text{is_subgraph} = \lambda H.\; \lambda G.\; (V_H \subseteq V_G) \;\wedge\; (E_H \subseteq E_G)" />

<DisplayMath formula="\text{induced} = \lambda G.\; \lambda S.\; \big(S,\; \{e \in E_G \mid e \subseteq S\}\big)" />

### Properties Inherited by Subgraphs

When you take a subgraph, some properties of the parent graph carry over and others can break:

**Connectivity is not preserved.** Even if G is connected, a subgraph can be disconnected. In the explorer above, step 3 shows a subgraph with vertices <InlineMath formula="\{A, B, C, D\}" /> and edges <InlineMath formula="\{A\text{-}B, C\text{-}D\}" /> — two disconnected pairs inside a connected G. An induced subgraph <InlineMath formula="G[S]" /> is connected only when the vertices in <InlineMath formula="S" /> form a connected piece of G.

**Degree can only decrease.** In a subgraph, every vertex has degree at most its degree in G — you can only *lose* edges, never gain them. Formally, for any subgraph <InlineMath formula="H" /> of <InlineMath formula="G" /> and any vertex <InlineMath formula="v \in V_H" />, <InlineMath formula="\deg_H(v) \leq \deg_G(v)" />.

**Bipartiteness is preserved by induced subgraphs.** If a graph has no odd cycles (we will define this property — *bipartiteness* — next lecture), then no induced subgraph can create one. Removing vertices can only destroy cycles, never create them.

In Python, these are single-operator expressions on edge sets:

```python
edges_union = edges_G1 | edges_G2        # G₁ ∪ G₂
edges_intersection = edges_G1 & edges_G2  # G₁ ∩ G₂
edges_difference = edges_G1 - edges_G2    # G₁ \ G₂
edges_sym_diff = edges_G1 ^ edges_G2      # G₁ △ G₂
```

The formalization pays off — the same algebra from LN10 works on this new structure.

### The Graph Complement

<DefinitionBox term="Graph Complement">

The **complement** of a graph <InlineMath formula="G = (V, E)" />, written <InlineMath formula="\overline{G}" />, has the same vertices but *exactly the edges G is missing*:

<DisplayMath formula="\overline{G} = \big(V,\; \{\{u,v\} \mid u,v \in V,\; u \neq v\} \setminus E\big)" />

</DefinitionBox>

Expressed as a lambda, the complement takes a graph and returns a new one. In LN10 we noted Python has no complement because there is no universal set; here the "universe" of possible edges is all 2-element subsets of <InlineMath formula="V" />:

<DisplayMath formula="\text{complement} = \lambda G.\; \big(V,\; \{\{u,v\} \mid u,v \in V,\; u \neq v\} \setminus E\big)" />

```python
all_possible = {frozenset({u, v}) for u in V for v in V if u != v}
complement_edges = all_possible - edges_G
```

## The Handshaking Lemma

Here is a beautiful connection between degrees and edges that links back to our combinatorics unit:

<DefinitionBox term="Handshaking Lemma">

In any undirected graph <InlineMath formula="G = (V, E)" />:

<DisplayMath formula="\sum_{v \in V} \deg(v) = 2|E|" />

Every edge contributes exactly 2 to the total degree — one for each endpoint.

</DefinitionBox>

<GraphVisualization
  nodes={[
    { id: "A", label: "A (deg 2)" },

    { id: "E", label: "E (deg 2)" },

    { from: "C", to: "D" },
    { from: "D", to: "A" },
    { from: "B", to: "D" },
    { from: "D", to: "E" },
  ]}
  interactive={true}

Sum of degrees: <InlineMath formula="2 + 3 + 2 + 3 + 2 = 12 = 2 \times 6 = 2|E|" />. It always works.

One immediate corollary: the number of vertices with **odd degree** is always **even** — since the total degree is even, there can't be an odd number of odd terms.

## Edge Contraction

We've seen how to combine graphs with set operations. Now let's define an operation that *modifies* a graph's structure: **edge contraction**.

<DefinitionBox term="Edge Contraction">

The **contraction** of edge <InlineMath formula="\{u, v\}" /> in graph <InlineMath formula="G = (V, E)" /> merges <InlineMath formula="u" /> and <InlineMath formula="v" /> into a single vertex <InlineMath formula="w" />, removing the contracted edge. Every other edge incident to <InlineMath formula="u" /> or <InlineMath formula="v" /> becomes incident to <InlineMath formula="w" />. **This operation preserves all edges** — if both <InlineMath formula="u" /> and <InlineMath formula="v" /> were adjacent to a common neighbor <InlineMath formula="x" />, the result has two parallel edges from <InlineMath formula="w" /> to <InlineMath formula="x" />, producing a **multigraph**.

</DefinitionBox>

## Vertex Cleaving

The inverse operation: **splitting** a vertex into two, duplicating its edges.

<DefinitionBox term="Vertex Cleaving">

**Vertex cleaving** splits a vertex <InlineMath formula="v" /> into two vertices <InlineMath formula="v_1, v_2" />. Every edge that was incident to <InlineMath formula="v" /> is **duplicated** — both <InlineMath formula="v_1" /> and <InlineMath formula="v_2" /> receive a copy, producing a **multigraph**. Optionally, an edge <InlineMath formula="\{v_1, v_2\}" /> is added.

</DefinitionBox>

**Dénes König (1884–1944)** wrote the first textbook on graph theory in 1936, formalizing many of the concepts we've covered today — components, connectivity, and operations like these. His work laid the foundation for the structural graph theory that would flourish in the following decades.

## Key Takeaways

<LectureNotes>

**Connectedness:**

- Connected graph: a path exists between every pair of vertices
- Disconnected graph: can be written as a union of graphs with disjoint vertex sets
- Connected components: maximal connected subgraphs — unique decomposition

**Cut Vertices, Bridges, and Bisection Width:**

- Cut vertex: removal disconnects the graph (single point of failure)
- Bridge: edge removal disconnects the graph
- Bisection width: minimum edges to cut to split a graph into two equal halves
- Real-world: network vulnerability, infrastructure resilience, data center design

**Subgraphs and Inherited Properties:**

- Subgraph: $V_H \subseteq V$, $E_H \subseteq E$
- Induced subgraph $G[S]$: all edges between vertices in $S$
- Connectivity is not preserved; degree can only decrease; bipartiteness is preserved by induced subgraphs

**Set Operations and Complement:**

- Set operations (union, intersection, difference, symmetric difference, complement) carry over from LN10
- Graph complement $\overline{G}$: same vertices, exactly the missing edges
- $G \cup \overline{G}$ produces a graph with every possible edge — formally defined next lecture

**Handshaking Lemma:**

- $\sum \deg(v) = 2|E|$ — every edge contributes 2 to total degree
- Number of odd-degree vertices is always even
- A graph with every possible edge has $\binom{n}{2} = n(n-1)/2$ edges (decaying options pattern from LN17)

**Edge Contraction and Vertex Cleaving:**

- Contraction: merge two adjacent vertices, preserving all edges — can produce multigraphs
- Cleaving: split a vertex into two, duplicating all incident edges — can produce multigraphs
- Dual operations: one merges, the other splits

**Lambda Calculus as a Tool:**

- is_cut_vertex, is_bridge: formalize "removal disconnects" using set difference
- is_subgraph, induced: express subgraph tests and construction via subset operations
- complement: builds the complement graph as a lambda
- Callback to LN19: using lambdas to express mathematical properties, not just compute values
