## Data Structures — CMSI 2120

**CMSI 2120 (Data Structures and Applications)** is the prerequisite to nearly every other CS course. It is where the *collections* of LN9–11 and the *functions* of LN14–15 grow up into engineered software components.

### Abstract Data Types ARE the LN24 Pattern

In LN24 we revealed that every topic in this course follows **Type → Instances → Operators**. CMSI 2120 takes that pattern and turns it into the central organizing principle of software engineering: an **Abstract Data Type (ADT)** is *literally* a type, a population of instances, and a contract of operators.

| ADT | Type | Instances | Operators |
| :--- | :--- | :--- | :--- |
| Stack | Sequence with LIFO discipline | `[]`, `[1, 2, 3]` | `push`, `pop`, `peek` |
| Queue | Sequence with FIFO discipline | `[]`, `[1, 2, 3]` | `enqueue`, `dequeue` |
| Set | Unordered, no duplicates | <InlineMath formula="\emptyset" />, <InlineMath formula="\{1, 2, 3\}" /> | <InlineMath formula="\cup, \cap, \setminus, \in" /> |
| Map | Function with finite domain | <InlineMath formula="\{a \mapsto 1, b \mapsto 2\}" /> | `get`, `put`, `delete` |
| Heap | Tree with heap-order property | min-heaps, max-heaps | `insert`, `extract-min` |

The right column of *every* row is set theory (LN9–10), function theory (LN14–15), or graph theory (LN23) wearing a software-engineering hat.

### Hash Tables: A Function That Tries to Be Injective

A hash table implements the Map ADT. Underneath, it is a function

<DisplayMath formula="h : \text{Keys} \longrightarrow \{0, 1, \ldots, B - 1\}" />

into a fixed set of <InlineMath formula="B" /> buckets. From LN15 we know functions need not be injective — and from the **pigeonhole principle (LN19)**, if there are more keys than buckets, *some* bucket must hold at least two keys. That is a **collision**, and it is mathematically unavoidable. Hash table design is the engineering art of making collisions *rare* in practice while accepting that they cannot be eliminated.

Insert keys one at a time. The "Normal Keys" mode shows a well-distributed hash function (insertions are amortized <InlineMath formula="O(1)" />); the "Adversarial Keys" mode shows what happens when an attacker chooses keys that *all* collide:

<HashTableVisualizer />

> **Key Insight:** When the hash function distributes well, the table behaves as a Map ADT with <InlineMath formula="O(n)" /> total work for <InlineMath formula="n" /> insertions. When all keys collide, the table degenerates into a linked list and the total cost rises to <InlineMath formula="O(n^2)" /> — exactly the combinatorial counting argument from LN17. The pigeonhole principle from LN19 guarantees this can happen; the *probabilistic* analysis (a future course's territory) bounds how often it does.

> **Side Note:** Tree-shaped data structures — binary search trees, AVL trees, red-black trees, B-trees — are the same trees from LN23, with extra invariants (balance, ordering) maintained by the operators. Heaps are trees with the heap-order property. The *graph-theoretic* fact that an <InlineMath formula="n" />-node tree has exactly <InlineMath formula="n - 1" /> edges (LN23) is what guarantees these structures use linear space.

## Algorithms and Analysis — CMSI 2130

**CMSI 2130 (Algorithms and Analysis)** is part of Computing Foundations — every CS major takes it. You will likely take it concurrently with or right after this course (CMSI 2120 is the formal prerequisite).

### BFS vs DFS

The two most fundamental graph traversal algorithms differ only in their data structure:

| Property | BFS | DFS |
| :--- | :--- | :--- |
| Data structure | Queue (FIFO) | Stack (LIFO) |
| Exploration pattern | Layer by layer | Deep before wide |
| Shortest path? | Yes (unweighted) | No |
| Memory | Proportional to width | Proportional to depth |

Step through both algorithms on the same graph. Switch tabs to see how one line of code — `queue.popleft()` vs `stack.pop()` — changes the entire traversal order:

<TraversalExplorer />

In Python, the implementations are nearly identical:

```python
from collections import deque

def bfs(graph, start):
    visited = set()
    queue = deque([start])
    order = []
    while queue:
        node = queue.popleft()   # FIFO — take from front
        if node not in visited:
            visited.add(node)
            order.append(node)
            for neighbor in graph[node]:
                if neighbor not in visited:
                    queue.append(neighbor)
    return order

def dfs(graph, start):
    visited = set()
    stack = [start]
    order = []
    while stack:
        node = stack.pop()       # LIFO — take from top
        if node not in visited:
            visited.add(node)
            order.append(node)
            for neighbor in graph[node]:
                if neighbor not in visited:
                    stack.append(neighbor)
    return order
```

> **Key Insight:** In CMSI 2130, you will analyze the *complexity* of these algorithms. BFS and DFS both run in <InlineMath formula="O(|V| + |E|)" /> time — the combinatorics from LN16–17 (counting operations) and the induction from LN25 (proving correctness) are the tools you will use.

### Sorting and Combinatorics

Sorting algorithm analysis uses the same "decaying options" pattern from LN17. Merge sort divides $n$ elements into halves, recursing <InlineMath formula="\log_2 n" /> times — its <InlineMath formula="O(n \log n)" /> complexity is proved by induction. The lower bound (no comparison sort can do better) is proved by contradiction — exactly the technique from LN25.

### Complexity Itself Follows the LN24 Pattern

Asymptotic analysis is its own little algebra — and it follows the LN24 *Type → Instances → Operators* pattern that has appeared in every block of this course:

- **Type:** asymptotic complexity classes
- **Instances:** <InlineMath formula="O(1),\ O(\log n),\ O(n),\ O(n \log n),\ O(n^2),\ O(n^3),\ O(2^n),\ O(n!)" />
- **Operators:** <InlineMath formula="O(f) + O(g) = O(\max(f, g))" />, <InlineMath formula="O(f) \cdot O(g) = O(f \cdot g)" />, <InlineMath formula="c \cdot O(f) = O(f)" />

## Computer Systems Organization — CMSI 2210

**CMSI 2210 (Computer Systems Organization)** is required for all CS majors and serves as the prerequisite to operating systems, databases, and networks. It is where the *abstract* number theory of LN5–7 collides with the *physical* reality of bytes, registers, and memory.

### Two's Complement IS Modular Arithmetic from LN7

How does a computer represent <InlineMath formula="-1" /> in 8 bits? It does *not* set aside one bit for the sign — that would be wasteful and would create two zeros (<InlineMath formula="+0" /> and <InlineMath formula="-0" />). Instead, it uses **two's complement**: the <InlineMath formula="2^k" /> bit patterns form the cyclic group <InlineMath formula="\mathbb{Z}/2^k\mathbb{Z}" /> from LN7, and the upper half is *relabeled* as the negative numbers.

Toggle between **4-bit, 8-bit, and 16-bit** to see how the trick scales. Drag the slider to sweep through every bit pattern, and watch the *same bits* get two different decimal interpretations — unsigned (just read the bits) and signed (the same bits with the sign bit flipped on means "subtract <InlineMath formula="2^k" />"):

<TwosComplementExplorer />

Why is this beautiful? Because **addition just works**. To compute <InlineMath formula="5 + (-3)" />, the hardware adds the unsigned bit patterns and lets the result wrap modulo <InlineMath formula="2^k" />. The same circuitry handles signed and unsigned arithmetic — there is no special "negative" hardware. **Subtraction is just modular negation** (LN7).

### Memory IS a Function

The next time you write `arr[i]` or dereference a pointer in C, remember that you are evaluating a function from LN15. Memory is literally a function:

<DisplayMath formula="\text{mem} : \text{Address} \longrightarrow \text{Byte}" />

with <InlineMath formula="\text{Address} = \mathbb{Z}/2^{64}\mathbb{Z}" /> on a 64-bit machine. Pointers are the *arguments*, dereferencing is *application*. Pointer arithmetic is arithmetic in the address space — modular when it overflows, just like LN7. Hexadecimal addresses (`0x7fff_ffff_e8a0`) are positional notation in base 16 from LN6.

> **Side Note:** Floating-point representation (IEEE 754) splits the 64 bits of a `double` into a sign bit, an 11-bit exponent (biased — modular arithmetic again), and a 52-bit mantissa. The whole format is a structured *encoding* — a function from real numbers (approximately) to bit patterns, designed so that arithmetic operators on the *bits* approximate arithmetic operators on the *reals*. Every numerical bug in scientific computing traces back to a place where this approximation leaks.

## Languages and Automata — CMSI 3801 / 3802

**CMSI 3801 and 3802 (Languages and Automata 1 & 2)** are required for all CS majors. They extend the lambda calculus from LN12–14 into a full theory of what computers can and cannot do.

### Finite Automata Are Directed Graphs

A finite automaton is a directed graph where nodes are **states**, edges are **transitions** labeled with input characters, and certain states are marked as **accepting**. The machine reads an input string character by character, following transitions. If it ends in an accept state, the string is accepted.

<DefinitionBox term="Deterministic Finite Automaton (DFA)">

A **DFA** is a 5-tuple <InlineMath formula="M = (Q, \Sigma, \delta, q_0, F)" /> where:

- <InlineMath formula="Q" /> is a finite set of **states** (LN9: sets)
- <InlineMath formula="\Sigma" /> is a finite **alphabet** (LN9: sets)
- <InlineMath formula="\delta : Q \times \Sigma \to Q" /> is the **transition function** (LN10: Cartesian product, LN15: functions)
- <InlineMath formula="q_0 \in Q" /> is the **start state** (LN9: element-of)
- <InlineMath formula="F \subseteq Q" /> is the set of **accepting states** (LN10: subset)

Every concept on the right comes from this course. **A DFA is just sets, products, and functions wired together.**

</DefinitionBox>

### Lambda Calculus → Computability

In LN12–14, we learned the lambda calculus. In CMSI 3801/3802, you will learn that Alonzo Church proved that lambda calculus can compute *everything* that is computable. Alan Turing independently proved the same thing for his **Turing machines** — DFAs from above, but with an *infinite tape* that the machine can both read and write to.

A Turing machine is a DFA (top diagram) coupled with an infinite tape that the head can both **read and write** (bottom strip). Each transition is labeled `read / write, move` — for instance, `0,1 / same, R` means "if you read either 0 or 1, write back the same symbol and move right." Step through the doubling machine below: in the **Scan** state the head walks rightward across the input, copying each digit; when it hits a blank it writes a 0 and enters **Halt**. The whole construction is just a directed graph from LN20 with a memory tape attached:

The **Church–Turing thesis** states that lambda calculus and Turing machines are equivalent: what one can compute, the other can — and *both* match every reasonable definition of computation that anyone has ever proposed. This thesis is the bedrock of theoretical CS.

The **halting problem** (LN25's contradiction proof) is proved in CMSI 3801/3802: no program can decide whether arbitrary programs terminate. The proof is exactly the contradiction argument from LN25 — assume such a program exists, construct a paradoxical input, derive a contradiction.

### Computability and Complexity — CMSI 5830 (Graduate)

For students who want to go deeper, **CMSI 5830 (Computability and Complexity)** is one of the required graduate courses. It extends the Turing machine above into the **complexity zoo**: classes like P (problems solvable in polynomial time), NP (verifiable in polynomial time), PSPACE, EXPTIME. The most famous open problem in CS — **does P = NP?** — lives here.

The map below is a tiny corner of that zoo. Each box is a *problem*; each region is a *complexity class*; each indigo arrow is a **polynomial-time reduction** (the LN25 technique). Click any problem to see what it means and how it connects to the others — then flip the toggle to imagine a world where someone proves **P = NP** and watch the orange region collapse into the green one:

<ComplexityZooMap />

A few things worth absorbing from the map:

- **Reductions form a directed graph (LN20).** The arrows from `3-SAT` to the other NP-complete problems are not metaphor — they are literal subroutine reductions. *If* you had a polynomial-time algorithm for `Vertex Cover`, you could *automatically* solve `3-SAT` in polynomial time by translating one input into the other. That is why the entire orange region rises and falls together.
- **Tiny syntactic changes can cross the P/NP-complete boundary.** `2-SAT` is in P, `3-SAT` is NP-complete. `Euler Cycle` (LN20) is in P, `Hamilton Cycle` is NP-complete. The boundary is razor-thin and not at all where intuition would predict — that is exactly the kind of result CMSI 5830 trains you to prove.
- **Some problems live in nobody's neighborhood.** `Graph Isomorphism` and `Integer Factoring` are in NP but neither known to be in P nor known to be NP-complete. RSA is built entirely on the bet that factoring stays in that limbo forever.

The whole field is built on:

- **Set theory (LN10)** — complexity classes are sets of languages, and inclusions like <InlineMath formula="\text{P} \subseteq \text{NP}" /> are subset relations
- **Combinatorial counting (LN17)** — running times are functions <InlineMath formula="f : \mathbb{N} \to \mathbb{N}" /> compared asymptotically
- **Reductions (LN25)** — the proof technique we used to show "if I can solve A, I can solve B" is the *exact* mechanism that defines NP-completeness

## Databases — CMSI 3520

**CMSI 3520 (Database Systems)** is one of the Systems and Architecture elective options. If you enjoyed LN9–11 (sets, operations, relations), databases are the direct application.

### Relational Algebra IS Set Operations

A relational database stores data in **tables** (also called **relations** — the same term from LN11). SQL queries are syntactic sugar for set operations.

<DefinitionBox term="Relational Schema and Relation">

Given attribute domains <InlineMath formula="A_1, A_2, \ldots, A_n" /> (each a set, LN9), a **relation** over this schema is a subset

<DisplayMath formula="R \subseteq A_1 \times A_2 \times \cdots \times A_n" />

of their Cartesian product (LN10). Each element of <InlineMath formula="R" /> is an <InlineMath formula="n" />-tuple — a *row*. Each <InlineMath formula="A_i" /> is a *column*.

This is the **exact** definition of a relation from LN11. Edgar Codd's 1970 invention of the relational model was the act of saying *"let's store all our data as LN11-style relations and query them with LN10 set operations."* That insight became Oracle, MySQL, PostgreSQL, and a multi-trillion-dollar industry.

</DefinitionBox>

The translation table from SQL to set theory:

## Digital Systems — EECE 2242

**EECE 2242 (Logic and Computer Design)** is required for all CS majors. It takes the Boolean algebra from LN5 and builds physical computers from it.

### Boolean Algebra IS Digital Logic

In LN5, we investigated the Booleans algebraically: <InlineMath formula="\mathbb{B} = \{0, 1\}" /> with addition (+), multiplication (×), and negation. In LN24's grand recap, we highlighted the isomorphism between Boolean algebra and classical logic.

<DefinitionBox term="Boolean Algebra (as a Structure)">

A **Boolean algebra** is a structure <InlineMath formula="(B, +, \cdot, \overline{\;\cdot\;}, 0, 1)" /> satisfying axioms — commutativity, associativity, distributivity, identity (<InlineMath formula="x + 0 = x" />, <InlineMath formula="x \cdot 1 = x" />), and complementation (<InlineMath formula="x + \overline{x} = 1" />, <InlineMath formula="x \cdot \overline{x} = 0" />).

This is the same *axiomatic* framing we used for ZFC in LN24: pick your axioms, derive everything else. Drop the law of excluded middle (<InlineMath formula="x + \overline{x} = 1" />) and you get a **Heyting algebra** — the algebraic semantics of intuitionistic logic from LN1. Different axioms, different algebra, different logic.

</DefinitionBox>

In EECE 2242, that isomorphism becomes *physical*:

| Boolean Algebra (LN5) | Logic Gate | Physical Implementation |
| :--- | :--- | :--- |
| <InlineMath formula="a \times b" /> (multiplication) | AND gate | Transistors in series |
| <InlineMath formula="a + b" /> (addition) | OR gate | Transistors in parallel |
| <InlineMath formula="\overline{a}" /> (negation) | NOT gate | Inverter circuit |
| <InlineMath formula="a \times b + \overline{a} \times c" /> | Combinational circuit | Wired gates |

Truth tables from LN2 become **circuit truth tables**. The same truth table you filled in for <InlineMath formula="P \land Q" /> is the specification for an AND gate. Boolean algebra simplification (DeMorgan's laws, distribution, absorption) is used to minimize the number of gates — fewer gates means cheaper, faster hardware.

Every computer ever built runs on Shannon's insight.

## Cryptography and Cryptanalysis — CMSI 2464

**CMSI 2464 (Cryptography and Cryptanalysis)** surveys both the classical and modern history of secret communication. The entire field is built on number theory from LN7 — modular arithmetic, prime factorization, modular inverses, and the Euclidean algorithm.

### Caesar Ciphers Are LN7 in Disguise

The simplest possible cipher — used by Julius Caesar himself — is the **shift cipher**. Encryption is *literally* a function of the form

<DisplayMath formula="E_k(x) = (x + k) \bmod 26" />

where each letter <InlineMath formula="x \in \{0, 1, \ldots, 25\}" /> is shifted by a key <InlineMath formula="k" />, wrapping around modulo 26. Decryption is just <InlineMath formula="D_k(y) = (y - k) \bmod 26" />. The set of valid keys is exactly the cyclic group <InlineMath formula="\mathbb{Z}/26\mathbb{Z}" /> from LN7.

Try it. Move the shift slider to change the key, type a plaintext into the encrypt slot to see its ciphertext, then type a ciphertext into the decrypt slot to recover its plaintext:

<CaesarPlayground />

Caesar ciphers are trivially broken (just try all 26 keys — that is the **brute force** attack from CMSI 2464). But the *structure* — encrypt by applying a function in <InlineMath formula="(\mathbb{Z}/n\mathbb{Z})" />, decrypt by applying its inverse — is the same structure that powers RSA below, AES, and every modern cipher. Bigger groups, harder to invert. That is the entire game.

### Hash Functions Are Deliberately Non-Injective

A **cryptographic hash function** like SHA-256 is a function

<DisplayMath formula="h : \{0, 1\}^* \longrightarrow \{0, 1\}^{256}" />

from arbitrary-length inputs to fixed-length outputs. From LN15 we know functions need not be injective — and from the **pigeonhole principle (LN19)**, a function with infinite domain and finite codomain *must* have collisions. Hash collisions are mathematically guaranteed to exist.

The cryptographic art is making them **computationally infeasible to find**. SHA-256 has <InlineMath formula="2^{256}" /> possible outputs — the *birthday paradox* (LN17 combinatorics) tells us a collision should be findable in roughly <InlineMath formula="2^{128}" /> attempts, which is more operations than every computer ever built has performed combined.

The RSA encryption algorithm — the foundation of secure internet communication — is built entirely on modular arithmetic from LN7. Step through the algorithm to see every piece in action:

Send the ciphertext down to the decrypt panel and watch the inverse operation recover the original. The security relies on the computational difficulty of **factoring** <InlineMath formula="n" /> back into its primes <InlineMath formula="p" /> and <InlineMath formula="q" />. For small numbers, factoring is easy. For 2048-bit numbers (the current standard), factoring is computationally intractable — a combinatorial explosion argument from LN17.

> **The Curry–Howard Correspondence:** The deepest result in PL foundations says **propositions ARE types** and **proofs ARE programs**. The implication <InlineMath formula="P \to Q" /> from LN1 *is* the function type <InlineMath formula="P \to Q" /> from LN15. Conjunction <InlineMath formula="P \land Q" /> *is* the pair type <InlineMath formula="P \times Q" /> from LN10. This is not analogy — it is literal isomorphism. It is the reason why a sufficiently expressive type system can encode any mathematical theorem, and why writing a program in such a system is *the same activity* as writing a proof.

<LectureNotes>

**Data Structures — CMSI 2120:**

- Abstract Data Types ARE the LN24 *Type → Instances → Operators* pattern
- Hash tables are functions <InlineMath formula="h : \text{Keys} \to \{0, \ldots, B-1\}" /> from LN15
- Pigeonhole principle (LN19) guarantees collisions are unavoidable
- Trees, heaps, and BSTs are LN23 trees with extra invariants

**Algorithms — CMSI 2130:**

- BFS and DFS are graph traversals from LN20, differing only in queue vs stack
- Both run in <InlineMath formula="O(|V| + |E|)" /> — analyzed with combinatorics and induction
- Dijkstra's correctness is proved by induction on settled vertices
- Sorting lower bounds are proved by contradiction
- Asymptotic complexity is its own algebra: <InlineMath formula="(O\text{-classes}, +, \cdot, \max)" />

**Computer Systems Organization — CMSI 2210:**

- Two's complement IS modular arithmetic from LN7 (in <InlineMath formula="\mathbb{Z}/2^k\mathbb{Z}" />)
- Memory is a function <InlineMath formula="\text{mem} : \text{Address} \to \text{Byte}" /> (LN15)
- Pointer arithmetic is arithmetic in the address space; hex addresses are LN6 base-16 notation

**Languages and Automata — CMSI 3801/3802:**

- DFAs are 5-tuples <InlineMath formula="(Q, \Sigma, \delta, q_0, F)" /> — pure sets, products, and functions
- Lambda calculus (LN12–14) and Turing machines compute the same things (Church–Turing)
- The halting problem (LN25) is proved by contradiction

**Databases — CMSI 3520:**

- A relation is formally <InlineMath formula="R \subseteq A_1 \times \cdots \times A_n" /> (LN10–11)
- SELECT/WHERE = set comprehension, UNION = ∪, INTERSECT = ∩, EXCEPT = ∖, JOIN = × + filter
- Python set comprehensions mirror SQL semantics exactly

**Digital Systems — EECE 2242:**

- Boolean algebra is the structure <InlineMath formula="(B, +, \cdot, \overline{\;\cdot\;}, 0, 1)" /> with axioms (LN5, LN24)
- AND = multiplication, OR = addition, NOT = negation
- Drop the law of excluded middle and you get a Heyting algebra (LN1's intuitionistic logic)

**Cryptography — CMSI 2464:**

- Caesar ciphers: <InlineMath formula="E_k(x) = (x + k) \bmod 26" /> — pure modular arithmetic from LN7
- Hash functions are deliberately non-injective; pigeonhole (LN19) guarantees collisions
- RSA uses modular arithmetic and prime factorization from LN7
- Birthday paradox (LN17) governs collision-finding cost

**Computability and Complexity — CMSI 5830 (Graduate):**

- Complexity classes are *sets of languages* (LN10), and inclusions like <InlineMath formula="\text{P} \subseteq \text{NP}" /> are subset relations
- Polynomial-time **reductions** (LN25) form a directed graph (LN20) over problems — the orange region in the Complexity Zoo rises and falls together
- `Euler` vs `Hamilton` and `2-SAT` vs `3-SAT` show the boundary between P and NP-complete is razor-thin and counterintuitive
- **P = NP?** is the open question that decides whether RSA (and most modern cryptography) survives

**Programming Languages Foundations — CMSI 5850 (Graduate):**

- The graduate flagship of the formal-theory thread; continues lambda + types from LN12–14
- Three semantic styles: **operational** (rule trees, what we showed), **denotational** (functions in math), **axiomatic** (Hoare triples)
- Operational rules are inference trees over AST nodes — a configuration <InlineMath formula="\langle e, \sigma, o \rangle" /> reduces step by step
- Curry–Howard: propositions ARE types, proofs ARE programs (literal isomorphism)
