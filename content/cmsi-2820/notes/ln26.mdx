### BFS vs DFS

The two most fundamental graph traversal algorithms differ only in their data structure:

| Property | BFS | DFS |
| :--- | :--- | :--- |
| Data structure | Queue (FIFO) | Stack (LIFO) |
| Exploration pattern | Layer by layer | Deep before wide |
| Shortest path? | Yes (unweighted) | No |
| Memory | Proportional to width | Proportional to depth |

Step through both algorithms on the same graph. Switch tabs to see how one line of code — `queue.popleft()` vs `stack.pop()` — changes the entire traversal order:

<TraversalExplorer />

In Python, the implementations are nearly identical:

```python
from collections import deque

def bfs(graph, start):
    visited = set()
    queue = deque([start])
    order = []
    while queue:
        node = queue.popleft()   # FIFO — take from front
        if node not in visited:
            visited.add(node)
            order.append(node)
            for neighbor in graph[node]:
                if neighbor not in visited:
                    queue.append(neighbor)
    return order

def dfs(graph, start):
    visited = set()
    stack = [start]
    order = []
    while stack:
        node = stack.pop()       # LIFO — take from top
        if node not in visited:
            visited.add(node)
            order.append(node)
            for neighbor in graph[node]:
                if neighbor not in visited:
                    stack.append(neighbor)
    return order
```

> **Key Insight:** In CMSI 2130, you will analyze the *complexity* of these algorithms. BFS and DFS both run in <InlineMath formula="O(|V| + |E|)" /> time — the combinatorics from LN16–17 (counting operations) and the induction from LN25 (proving correctness) are the tools you will use.

### Sorting and Combinatorics

Sorting algorithm analysis uses the same "decaying options" pattern from LN17. Merge sort divides $n$ elements into halves, recursing <InlineMath formula="\log_2 n" /> times — its <InlineMath formula="O(n \log n)" /> complexity is proved by induction. The lower bound (no comparison sort can do better) is proved by contradiction — exactly the technique from LN25.

## Languages and Automata — CMSI 3801 / 3802

**CMSI 3801 and 3802 (Languages and Automata 1 & 2)** are required for all CS majors. They extend the lambda calculus from LN12–14 into a full theory of what computers can and cannot do.

### Finite Automata Are Directed Graphs

A finite automaton is a directed graph where nodes are **states**, edges are **transitions** labeled with input characters, and certain states are marked as **accepting**. The machine reads an input string character by character, following transitions. If it ends in an accept state, the string is accepted.

### Lambda Calculus → Computability

In LN12–14, we learned the lambda calculus. In CMSI 3801/3802, you will learn that Alonzo Church proved that lambda calculus can compute *everything* that is computable. Alan Turing independently proved the same thing for his Turing machines. The **Church–Turing thesis** states these are equivalent — what lambdas can compute, machines can compute, and vice versa.

The halting problem (LN25's contradiction proof) is proved in this course: no program can decide whether arbitrary programs terminate. This impossibility result is the cornerstone of theoretical CS.

> **Side Note:** For graduate students, **CMSI 583 (Computability and Complexity)** goes deeper — studying which problems are efficiently solvable and which are inherently hard. **CMSI 585 (Programming Language Foundations)** continues the type theory and lambda calculus thread from LN4–5 and LN12–14, formalizing type systems and proving programming languages correct.

### Relational Algebra IS Set Operations

A relational database stores data in **tables** (also called **relations** — the same term from LN11). SQL queries are syntactic sugar for set operations:

| SQL | Set Operation | LN Reference |
| :--- | :--- | :--- |
| `SELECT ... WHERE` | Set comprehension / filter | LN10: set-builder notation |
| `UNION` | <InlineMath formula="A \cup B" /> | LN10: set union |
| `INTERSECT` | <InlineMath formula="A \cap B" /> | LN10: set intersection |
| `EXCEPT` | <InlineMath formula="A \setminus B" /> | LN10: set difference |
| `JOIN` | <InlineMath formula="A \times B" /> followed by filter | LN10: Cartesian product |

> **Key Insight:** When you write SQL, you are writing set operations. The database engine optimizes them — deciding whether to compute the Cartesian product first or filter first, which join algorithm to use, which indexes to leverage. But the *semantics* — what the query *means* — is pure set theory from LN9–10.

```python
students = {
    (1, "Alice", "CS"), (2, "Bob", "Math"),
    (3, "Carol", "CS"), (4, "Dave", "Physics")
}

cs_students = {s for s in students if s[2] == "CS"}
```

Python's set comprehension IS the WHERE clause. The curly braces, the `for ... in`, the `if` condition — this is set-builder notation in Python syntax.

## Digital Systems — EECE 2242

**EECE 2242 (Logic and Computer Design)** is required for all CS majors. It takes the Boolean algebra from LN5 and builds physical computers from it.

### Boolean Algebra IS Digital Logic

In LN5, we investigated the Booleans algebraically: <InlineMath formula="\mathbb{B} = \{0, 1\}" /> with addition (+), multiplication (×), and negation. In LN24's grand recap, we highlighted the isomorphism between Boolean algebra and classical logic.

In EECE 2242, that isomorphism becomes *physical*:

| Boolean Algebra (LN5) | Logic Gate | Physical Implementation |
| :--- | :--- | :--- |
| <InlineMath formula="a \times b" /> (multiplication) | AND gate | Transistors in series |
| <InlineMath formula="a + b" /> (addition) | OR gate | Transistors in parallel |
| <InlineMath formula="\overline{a}" /> (negation) | NOT gate | Inverter circuit |
| <InlineMath formula="a \times b + \overline{a} \times c" /> | Combinational circuit | Wired gates |

Truth tables from LN2 become **circuit truth tables**. The same truth table you filled in for <InlineMath formula="P \land Q" /> is the specification for an AND gate. Boolean algebra simplification (DeMorgan's laws, distribution, absorption) is used to minimize the number of gates — fewer gates means cheaper, faster hardware.

<GateExplorer />

> **Historical Note:** **George Boole (1815–1864)** invented Boolean algebra in 1854 as a way to formalize logic. It was purely theoretical for nearly a century. In 1937, **Claude Shannon** showed in his master's thesis that Boolean algebra could describe electrical switching circuits — connecting abstract math to physical engineering. Every computer ever built runs on Shannon's insight.

While LMU does not have a dedicated cryptography course, these ideas appear in **CMSI 3550 (Networks and Internets)** and in graduate electives.

### RSA and Modular Arithmetic

The RSA encryption algorithm — the foundation of secure internet communication — is built entirely on modular arithmetic from LN7.

Step through the algorithm with small numbers to see every piece of modular arithmetic in action. Then try encrypting your own message:

<RSAStepper />

The security relies on the computational difficulty of **factoring** $n$ back into $p$ and $q$. For small numbers, factoring is easy. For 2048-bit numbers (the current standard), factoring is computationally intractable — a combinatorial explosion argument from LN17.

> Every time you visit a website with `https://`, RSA (or its successor, elliptic curve cryptography) is protecting your data using the modular arithmetic you learned in LN7.
